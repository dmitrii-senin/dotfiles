# Build/Debug/Git/Performance Tooling Topic Bank

## beginner

### Git signs
Gitsigns.nvim shows git diff indicators in the sign column: added lines get a green bar, modified lines get a blue bar, and deleted lines get a red triangle. Your config uses `]h`/`[h` to navigate between hunks (changed regions), which is the fastest way to review what you have modified in a file. The `ih` text object (in operator-pending and visual modes) selects the current hunk, letting you operate on changed code with standard Vim operators. Gitsigns works asynchronously and updates signs in real time as you edit, with no perceptible lag even on large files.
**Key concepts:** sign column indicators, hunk navigation, ]h/[h, ih text object, added/modified/deleted signs, asynchronous updates
**Tip:** Use `[h` and `]h` before committing to quickly review every change in a file without leaving the editor. Combine with `<Leader>gp` (preview hunk) to see the exact diff for each hunk inline, giving you a lightweight code review workflow without switching to a separate diff tool.
**Tool anchor:** `]h`/`[h` for hunk navigation; `<Leader>gp` for hunk preview; `ih` text object for hunk selection in visual/operator-pending mode
**Drill:** Make three separate changes in different parts of a file. Use `]h` to jump to each hunk in sequence, preview each with `<Leader>gp`, and verify the sign column shows the correct indicator type (add vs modify vs delete). Then use `vih` to visually select a hunk and confirm it selects exactly the changed lines.
**Tags:** gitsigns, sign-column, hunk-navigation, diff-indicators, text-object

### Git staging from Neovim
Your gitsigns config lets you stage and unstage hunks directly from the editor without switching to a terminal or external tool. `<Leader>gs` stages the hunk under the cursor, `<Leader>gr` resets (discards) it, `<Leader>gu` undoes the last stage operation, and `<Leader>gS`/`<Leader>gR` stage/reset the entire buffer. In visual mode, `<Leader>gs` and `<Leader>gr` operate on the selected lines, enabling partial hunk staging. This gives you the precision of `git add -p` with the speed of staying in your editor. Staged hunks change their sign column indicator to show they are already in the index.
**Key concepts:** stage_hunk, reset_hunk, undo_stage_hunk, stage_buffer, visual-mode partial staging, git index
**Tip:** Partial hunk staging in visual mode is powerful for separating concerns: if a hunk contains both a bug fix and a formatting change, select just the fix lines with `V` and stage them with `<Leader>gs`, leaving the formatting for a separate commit. This is faster than `git add -p` because you can see the code context directly.
**Tool anchor:** `<Leader>gs` to stage hunk; `<Leader>gr` to reset hunk; `<Leader>gu` to undo stage; `<Leader>gS`/`<Leader>gR` for full buffer; visual `<Leader>gs` for partial staging
**Drill:** Make two logically separate changes in the same function. Enter visual mode (`V`), select only the lines for the first change, and stage them with `<Leader>gs`. Verify with `<Leader>gd` (diffthis) that only the first change is staged. Then use `<Leader>gg` (LazyGit) to commit just the staged change, leaving the second change unstaged.
**Tags:** gitsigns, staging, partial-staging, reset-hunk, undo-stage, git-index

### LazyGit integration
LazyGit is a terminal UI for git that provides visual staging, commit, push, branch management, and interactive rebase in a single interface. Your config maps `<Leader>gg` to open LazyGit in a full-screen overlay via lazygit.nvim. Inside LazyGit, you navigate with `hjkl`, stage files with `space`, commit with `c`, push with `P`, and view diffs for any file or commit. The Telescope integration (`require("telescope").load_extension("lazygit")`) lets you browse LazyGit repositories from Telescope. LazyGit handles complex git operations (interactive rebase, cherry-pick, stash management) more safely than raw command-line git.
**Key concepts:** LazyGit overlay, visual staging, commit workflow, push, branch management, interactive rebase, Telescope integration
**Tip:** LazyGit's diff view is excellent for reviewing changes before committing. Press `enter` on any file to see its diff, and use `space` to stage/unstage individual files. For interactive rebase, switch to the commits panel with `2`, select a range of commits, and press `e` to edit, `s` to squash, or `d` to drop -- all with visual feedback before applying.
**Tool anchor:** `<Leader>gg` to open LazyGit; navigate panels with `1-5`; `space` to stage; `c` to commit; `P` to push; `q` to quit
**Drill:** Make several changes across multiple files, open LazyGit with `<Leader>gg`, stage specific files with `space`, write a commit message with `c`, and push with `P`. Then practice an interactive rebase: switch to the commits panel, select two recent commits, and squash them together. Exit with `q` and verify the result in the editor.
**Tags:** lazygit, git-ui, staging, commit, push, interactive-rebase, Telescope

### Terminal usage
Your toggleterm.nvim config provides a floating terminal toggled with `<Leader>tt`, dismissed with `<Esc><Esc>`. The floating terminal appears as an overlay centered on the screen, ideal for quick commands without leaving your editing context. Terminal mode in Neovim uses a special insert-like mode where most keys go to the terminal process; `<Esc><Esc>` is mapped to close/hide the terminal via `<Cmd>ToggleTerm<CR>`. The terminal state persists between toggles, so long-running processes (build watch, test runner, development server) continue in the background when you dismiss the terminal.
**Key concepts:** toggleterm, floating terminal, terminal mode, persistent state, Esc-Esc exit, background processes
**Tip:** Use the floating terminal for quick one-off commands (build, test, git operations) and keep it open for interactive work. Since the terminal state persists, you can start a build with `<Leader>tt`, type `cmake --build build`, dismiss with `<Esc><Esc>`, continue editing, and toggle back to check the result -- the build output is still there.
**Tool anchor:** `<Leader>tt` to toggle floating terminal; `<Esc><Esc>` to dismiss; terminal persists between toggles
**Drill:** Open a floating terminal with `<Leader>tt`, run `ls -la`, dismiss with `<Esc><Esc>`, edit some code, then toggle the terminal back and verify the `ls` output is still visible. Start a long-running command like `cargo build`, dismiss the terminal, and toggle back periodically to check progress.
**Tags:** toggleterm, terminal, floating, persistent, terminal-mode, background

### Running shell commands
Neovim provides several ways to run shell commands: `:!cmd` runs a command and shows output, `vim.system({'cmd', 'arg'})` runs commands programmatically from Lua, `:read !cmd` inserts command output into the buffer, and `:write !cmd` pipes the buffer through a command. Your `grepprg` is set to `rg --vimgrep` for fast recursive search via `:grep`. The `:!` command blocks Neovim until completion, while `vim.system` with a callback runs asynchronously. For captured output that you want to navigate, `:cexpr system('cmd')` populates the quickfix list.
**Key concepts:** :! command, vim.system, :read !, :write !, :grep, output capture, quickfix from command, async execution
**Tip:** Use `:read !date` to insert the current date, `:read !curl -s url` to fetch content into a buffer, and `:%!python3 -m json.tool` to format the entire buffer as pretty-printed JSON. For C++ development, `:read !xxd` on a binary file gives you a hex editor, and `:%!xxd -r` converts it back.
**Tool anchor:** `:!cmd` for one-shot commands; `:read !cmd` for output insertion; `:%!cmd` for buffer filtering; `:grep pattern` for ripgrep search
**Drill:** Open a JSON file and format it with `:%!python3 -m json.tool`. Then use `:grep` to search your project for a function name and navigate results with `]q`/`[q`. Try `:cexpr system('cmake --build build 2>&1')` to capture build errors into the quickfix list.
**Tags:** shell-commands, grep, quickfix, buffer-filter, vim-system, output-capture

### Startup time basics
Neovim startup time directly affects your workflow responsiveness. The `nvim --startuptime /tmp/startup.log` flag writes a detailed timing log showing how long each plugin, script, and configuration file takes to load. Your config uses lazy.nvim for plugin management, which defers loading plugins until they are needed (via `event`, `cmd`, `ft`, or `keys` triggers). Most of your plugins are lazy-loaded: lspconfig loads on file open, gitsigns on `BufReadPre`, conform on `BufReadPre`/`BufNewFile`. Mason is deferred with `cmd` trigger, and DAP loads only when you press a `<Leader>D` key. Understanding what loads eagerly vs lazily is key to keeping startup under 50ms.
**Key concepts:** --startuptime, lazy.nvim, lazy-loading triggers (event/cmd/ft/keys), eager vs deferred, startup profiling
**Tip:** Run `nvim --startuptime /tmp/startup.log +q && sort -k2 -n /tmp/startup.log | tail -20` to find the 20 slowest items during startup. If a plugin you rarely use is loading eagerly, add appropriate lazy-loading triggers. Your treesitter config with `lazy = false` loads eagerly -- this is correct since syntax highlighting is needed immediately.
**Tool anchor:** `nvim --startuptime /tmp/startup.log` for profiling; `:Lazy` (`<Leader>L`) for plugin manager UI; `:Lazy profile` for load times
**Drill:** Run `nvim --startuptime /tmp/startup.log +q` and read the log to identify the three slowest items. Open Neovim and run `:Lazy` to see the plugin list and which are loaded. Compare the total startup time with and without your heaviest plugin by temporarily disabling it in your config.
**Tags:** startup-time, lazy-loading, lazy-nvim, profiling, performance, plugin-management

### :make and build output
The `:make` command runs `makeprg` (default: `make`) and parses its output through `errorformat` to populate the quickfix list with file:line:column entries. Your config provides `[q`/`]q` to navigate quickfix entries and `<Leader>lq` to open them in Trouble. For C++ projects, setting `makeprg` to `cmake --build build` or `ninja -C build` integrates your build directly. The errorformat for GCC/Clang is built-in (`%f:%l:%c: %m`), so compiler errors, warnings, and notes all become navigable quickfix entries. `:make` blocks Neovim during the build; for async builds, use `:make` in combination with the terminal or an async build plugin.
**Key concepts:** makeprg, errorformat, quickfix list, :make, build integration, [q/]q navigation, Trouble quickfix
**Tip:** Set up per-filetype makeprg in your config or a project-local `.nvim.lua`: for C++ use `cmake --build build`, for Rust use `cargo build`, for Python use `python3 -m pytest`. This way `:make` always does the right thing regardless of which file you are editing.
**Tool anchor:** `:make` to build; `[q`/`]q` for error navigation; `<Leader>lq` for Trouble quickfix view; `:set makeprg?` to check current setting
**Drill:** In a C++ project, set `makeprg` to your CMake build command and introduce a deliberate compile error. Run `:make`, navigate to the error with `]q`, fix it, and run `:make` again to verify the fix. Open the quickfix in Trouble with `<Leader>lq` to see all warnings and notes alongside errors.
**Tags:** make, quickfix, errorformat, build, compiler-errors, Trouble

### Persistent undo
Your config enables persistent undo with `opt.undofile = true` and `opt.undolevels = 10000`, meaning Neovim saves your undo history to disk and restores it when you reopen a file. This lets you undo changes from days or weeks ago, even after closing and reopening Neovim. Undo files are stored in `~/.local/state/nvim/undo/` by default (following XDG conventions). The undo tree is a branching structure, not a linear list -- if you undo several changes and then make a new change, the undone changes are preserved on a separate branch accessible with `g-`/`g+` or `:earlier`/`:later`.
**Key concepts:** undofile, undolevels, undo tree, branching undo, g-/g+, :earlier/:later, XDG undo directory
**Tip:** Use `:earlier 10m` to jump to the buffer state from 10 minutes ago, or `:earlier 3f` to jump back 3 file writes. This is invaluable when you realize a refactoring went wrong and want to compare with the state from before you started. The undotree plugin provides a visual tree browser if you need to explore branches.
**Tool anchor:** `u` to undo; `<C-r>` to redo; `g-`/`g+` for undo branches; `:earlier 10m` for time-based undo; `:undolist` to see branches
**Drill:** Open a file, make several changes, save, close, and reopen it. Verify that `u` can undo past the last save. Then make some changes, undo several steps, make a different change (creating a branch), and use `g-`/`g+` to navigate between undo branches. Try `:earlier 5m` to jump back in time.
**Tags:** undo, undofile, persistent-undo, undo-tree, earlier-later, branching-undo

## intermediate

### Git blame and diff
Your gitsigns config provides `<Leader>gb` for inline blame (shows the commit author, date, and message for the current line in a floating window with `full = true`) and `<Leader>gB` to toggle persistent current-line blame as virtual text. For diffing, `<Leader>gd` opens a side-by-side diff of the current file against the git index (staged version), and `<Leader>gD` diffs against the previous commit (`~`). Neovim's built-in diff mode (`vimdiff`) highlights additions, deletions, and changes with configurable fill characters (your config uses `diff = "/"` for deleted lines). Comparing specific branches requires `:Gitsigns diffthis origin/main` or using LazyGit's diff viewer.
**Key concepts:** blame_line, toggle_current_line_blame, diffthis, diff against index vs HEAD~, diff mode, branch comparison
**Tip:** Use `<Leader>gb` on a puzzling line to see who wrote it, when, and the commit message explaining why. If the blame shows a large commit, open LazyGit with `<Leader>gg` and navigate to that commit to see the full change context. For comparing branches, `:Gitsigns diffthis origin/main` is faster than switching to a terminal.
**Tool anchor:** `<Leader>gb` for blame; `<Leader>gB` for toggle line blame; `<Leader>gd` for diff vs index; `<Leader>gD` for diff vs HEAD~
**Drill:** Open a file with a long history. Use `<Leader>gb` on several lines to read their blame information. Toggle persistent blame with `<Leader>gB` and scroll through the file to see commit info on every line. Then use `<Leader>gd` to view the current unstaged changes as a side-by-side diff, and `<Leader>gD` to compare against the previous commit.
**Tags:** gitsigns, blame, diff, diffthis, git-history, branch-comparison

### Git hunk operations
Beyond basic staging, your gitsigns config supports a complete hunk workflow. Preview (`<Leader>gp`) shows the hunk diff inline without opening a full diff view. Stage (`<Leader>gs`) adds the hunk to the index, and undo-stage (`<Leader>gu`) reverses it -- this is a safe undo that only affects the last stage operation. Reset (`<Leader>gr`) discards the hunk entirely, reverting to the index version. In visual mode, `<Leader>gs` and `<Leader>gr` operate on the selected line range, enabling surgical partial-hunk staging. The `ih` text object selects the entire hunk for use with any operator (`dih` to delete a hunk, `yih` to yank it).
**Key concepts:** preview hunk, stage hunk, undo stage, reset hunk, visual partial staging, ih text object, operator integration
**Tip:** The workflow for crafting clean commits: navigate hunks with `]h`/`[h`, preview each with `<Leader>gp` to decide if it belongs in this commit, stage with `<Leader>gs` or skip. For hunks that mix two concerns, enter visual mode, select the relevant lines, and use `<Leader>gs` to stage only those lines. Use `<Leader>gu` if you stage something by mistake.
**Tool anchor:** `<Leader>gp` preview; `<Leader>gs` stage; `<Leader>gu` undo; `<Leader>gr` reset; visual `<Leader>gs`/`<Leader>gr` for partial ops; `ih` text object
**Drill:** Make four changes to a file: two that belong in one commit and two in another. Preview each hunk with `<Leader>gp`. Stage only the first two with `<Leader>gs`, accidentally stage a third, undo with `<Leader>gu`, then commit via `<Leader>gg`. Stage and commit the remaining hunks in a second commit. Verify the two commits have clean, focused diffs.
**Tags:** gitsigns, hunk-operations, staging, preview, partial-staging, clean-commits

### Lazy loading strategies
Lazy.nvim supports four primary lazy-loading triggers: `event` (Neovim events like `BufReadPre`, `VeryLazy`), `cmd` (Vim commands like `:LazyGit`), `ft` (filetype detection), and `keys` (keymap activation). Your config demonstrates each pattern: gitsigns uses `event = {"BufReadPre", "BufNewFile"}` to load when a file is opened, mason uses `cmd` to load only when `:Mason` is invoked, lazygit uses both `cmd` and `keys` for dual activation paths, and DAP uses `keys` to load only when a `<Leader>D*` mapping is pressed. The `:Lazy profile` command shows load times and trigger sources. Measured startup impact is the key metric -- load what you need, when you need it.
**Key concepts:** lazy-loading triggers, event/cmd/ft/keys, VeryLazy, startup impact, :Lazy profile, load order, plugin dependencies
**Tip:** The `VeryLazy` event fires after the UI renders, making it ideal for plugins that are needed soon but not at startup (like mason-tool-installer with its `start_delay = 3000`). Avoid using `lazy = false` unless the plugin must be available immediately (treesitter for syntax, catppuccin for colorscheme). Every `lazy = false` plugin adds directly to startup time.
**Tool anchor:** `:Lazy` (`<Leader>L`) for plugin UI; `:Lazy profile` for load timing; check `lazy`, `event`, `cmd`, `keys`, `ft` fields in plugin specs
**Drill:** Open `:Lazy` and note which plugins are loaded vs not loaded. Press `<Leader>Db` to trigger a DAP keymap and check `:Lazy` again -- DAP plugins should now be loaded. Run `:Lazy profile` to see the load time for each plugin and identify which took the longest.
**Tags:** lazy-loading, lazy-nvim, startup, event, cmd, keys, profiling

### Plugin profiling
The `:Lazy profile` view shows exactly when each plugin loaded, how long its `config` function took, and what triggered the load. Sort by load time to identify bottlenecks. Beyond lazy.nvim's built-in profiling, you can trace Lua module loading with `require` interception to find expensive initialization. Common bottlenecks include: LSP server startup (clangd indexing), treesitter parser compilation on first use, Mason tool installation checks, and heavy `config` functions that do synchronous I/O. Profile regularly when adding plugins to catch regressions.
**Key concepts:** :Lazy profile, config function timing, require tracing, bottleneck identification, startup regression, module loading
**Tip:** If `:Lazy profile` shows a plugin's config function taking more than 10ms, investigate what it does at init time. Common culprits: `require("telescope")` chains loading multiple modules, synchronous HTTP checks, or file system scans. Move expensive initialization into autocommands or lazy-load the expensive parts on first use.
**Tool anchor:** `:Lazy profile` for timing; `:Lazy` for plugin status; `nvim --startuptime` for full startup analysis; `:lua print(vim.inspect(package.loaded))` to see loaded modules
**Drill:** Run `:Lazy profile` and identify the three plugins with the longest load times. For each, read the `config` function in your plugin spec and determine what makes it slow. Try deferring one plugin's load from `lazy = false` to `event = "VeryLazy"` and measure the startup time improvement with `nvim --startuptime`.
**Tags:** profiling, lazy-nvim, startup, config-timing, bottleneck, require-tracing

### Autocommand performance
Autocommands that fire on frequent events (`CursorHold`, `CursorMoved`, `TextChanged`, `InsertLeave`) can accumulate significant overhead. Your config creates several autocommand groups: `custom-lsp-attach` for LSP setup, `custom-lsp-highlight` for document highlight on `CursorHold`/`CursorMoved`, `custom-treesitter-highlight` for syntax on `FileType`, and nvim-lint triggers on `BufWritePost`/`BufReadPost`/`InsertLeave`. Each autocommand should be lightweight; expensive operations should be debounced or deferred. The `updatetime = 200` setting (how quickly `CursorHold` fires after the cursor stops moving) balances responsiveness with overhead.
**Key concepts:** autocommand events, CursorHold, CursorMoved, augroup, debouncing, updatetime, performance overhead, event frequency
**Tip:** The `CursorMoved` event fires on every cursor movement and can cause lag if the callback is expensive. Your LSP `clear_references` on `CursorMoved` is lightweight (just clears highlights), but be cautious adding more `CursorMoved` handlers. If you need to do expensive work on cursor movement, debounce it with a timer: `vim.defer_fn(function() ... end, 100)` and cancel the previous timer on each invocation.
**Tool anchor:** `:autocmd` to list all autocommands; `:au CursorHold` to see CursorHold handlers; `vim.api.nvim_get_autocmds({event = "CursorHold"})` for Lua inspection
**Drill:** List all autocommands for the `CursorHold` event with `:au CursorHold` and count how many handlers are registered. Then open a large file and observe whether cursor hold operations (document highlight, diagnostics) cause any perceptible delay. If they do, try increasing `updatetime` to 500 temporarily and feel the difference.
**Tags:** autocommands, CursorHold, performance, debouncing, updatetime, augroup

### DAP architecture
The Debug Adapter Protocol (DAP) decouples the editor (client) from the debugger (adapter), similar to how LSP decouples editors from language analysis. Your config uses two adapters: codelldb for C/C++/Rust (a server-type adapter that listens on a dynamic port, started by Mason from `~/.local/share/nvim/mason/bin/codelldb`) and debugpy for Python (initialized by dap-python). Each adapter has configurations that specify how to launch or attach to a program. The `launch` request starts a new process; `attach` connects to an already-running process. nvim-dap orchestrates the protocol, and dap-ui provides the visual interface with scopes, watch, REPL, breakpoints, and stack trace panels.
**Key concepts:** DAP protocol, adapter types (server/executable), launch vs attach, configurations, dap-ui panels, Mason-managed adapters
**Tip:** Understanding launch vs attach is critical: `launch` starts the program under debugger control (your current config), while `attach` connects to a running process by PID. For debugging a live market data handler, you need an `attach` configuration: add `{name = "Attach", type = "codelldb", request = "attach", pid = function() return tonumber(vim.fn.input("PID: ")) end}` to your DAP configurations.
**Tool anchor:** `<Leader>Dc` to start a session; `<Leader>Du` to toggle dap-ui; `:lua print(vim.inspect(require("dap").configurations.cpp))` to see configurations
**Drill:** Examine your DAP configuration by running `:lua print(vim.inspect(require("dap").configurations.cpp))` and understanding each field. Start a debug session with `<Leader>Dc`, observe the dap-ui panels opening automatically (via the `event_initialized` listener), and identify each panel: scopes, watches, breakpoints, stack trace, REPL, and console.
**Tags:** DAP, debug-adapter-protocol, codelldb, debugpy, launch-attach, dap-ui, Mason

### DAP workflow
A typical debug session follows this flow: set breakpoints with `<Leader>Db`, start the session with `<Leader>Dc` (continue), and when execution hits a breakpoint, use `<Leader>Do` (step over), `<Leader>Di` (step into), `<Leader>DO` (step out) to navigate execution. The dap-ui panels show local variables (scopes), watched expressions, the call stack, and active breakpoints. The REPL (`<Leader>Dr`) lets you evaluate arbitrary expressions in the current scope. `<Leader>DB` sets conditional breakpoints that only trigger when a given expression is true. `<Leader>Dl` re-runs the last debug configuration, and `<Leader>Dt` terminates the session.
**Key concepts:** breakpoints, continue, step over/into/out, scopes, watch, REPL evaluation, conditional breakpoints, run last, terminate
**Tip:** Use conditional breakpoints (`<Leader>DB`) extensively when debugging message handlers: instead of breaking on every message and manually continuing until you hit the interesting case, set `templateId == 32` as the condition. For data-dependent bugs, use the REPL (`<Leader>Dr`) to inspect complex expressions like `order_book.best_bid()` without adding print statements.
**Tool anchor:** `<Leader>Db` breakpoint; `<Leader>DB` conditional; `<Leader>Dc` continue; `<Leader>Do/Di/DO` step; `<Leader>Dr` REPL; `<Leader>Du` UI; `<Leader>Dt` terminate
**Drill:** Set up a debug session for a program with a loop. Place a regular breakpoint at the loop entry, then replace it with a conditional breakpoint (`<Leader>DB`) that triggers only when the loop counter exceeds 50. Step through a few iterations with `<Leader>Do`, inspect a variable in the REPL with `<Leader>Dr`, add it to the watch panel in dap-ui, and observe it update as you step.
**Tags:** DAP, debugging, breakpoints, stepping, REPL, watch, conditional-breakpoints

## advanced

### Git conflict resolution
Git merge conflicts produce conflict markers (`<<<<<<<`, `=======`, `>>>>>>>`) that need manual resolution. Neovim's built-in diff mode (`:diffthis` on two splits) provides a basic 2-way diff, but 3-way merges require a dedicated tool. Diffview.nvim adds a file-tree panel with merge status indicators and a 3-way diff layout showing LOCAL/BASE/REMOTE versions. Your gitsigns config detects conflict markers and shows them as modified hunks. The resolution workflow is: open the conflicted file, understand both sides, edit to the desired result, stage with `<Leader>gs`, and mark resolved. For complex merges, LazyGit's merge panel provides visual conflict resolution.
**Key concepts:** conflict markers, 3-way diff, LOCAL/BASE/REMOTE, diffview.nvim, merge resolution, vimdiff, conflict detection
**Tip:** Configure Neovim as your git mergetool with `git config merge.tool nvimdiff` and `git config mergetool.nvimdiff.layout "LOCAL,BASE,REMOTE / MERGED"`. This opens a 4-pane layout when `git mergetool` is invoked: three read-only panes showing each version and one editable pane for the result. Use `]c`/`[c` to jump between conflict regions and `:diffget LOCAL` or `:diffget REMOTE` to accept changes from either side.
**Tool anchor:** `]h`/`[h` to navigate conflicts in gitsigns; `<Leader>gg` for LazyGit merge view; `:diffthis`/`:diffoff` for manual diff mode
**Drill:** Create a merge conflict intentionally by modifying the same line on two branches. Attempt the merge, open the conflicted file in Neovim, and navigate conflict markers with `]h`/`[h`. Resolve the conflict by editing the merged result, then stage with `<Leader>gs`. Open LazyGit to verify the resolution and complete the merge commit.
**Tags:** merge-conflict, 3-way-diff, diffview, conflict-resolution, mergetool, vimdiff

### DAP advanced
Beyond basic breakpoints, DAP supports logpoints (print a message without stopping), conditional breakpoints with hit counts (break after N hits), exception breakpoints (break on thrown/uncaught exceptions), and data breakpoints (break when a memory address changes -- hardware watchpoints). Codelldb supports LLDB expressions in conditional breakpoints, including calling functions and accessing struct members. For remote debugging, codelldb can connect to a remote `lldb-server` or `gdbserver` via `attach` requests with a host:port target. Exception breakpoints are configured per-adapter and catch C++ `throw`, Python `raise`, and Rust `panic!`.
**Key concepts:** logpoints, hit-count breakpoints, exception breakpoints, data breakpoints, remote debugging, LLDB expressions

### Performance checklist
Several Neovim settings directly affect editor performance: `updatetime = 200` controls `CursorHold` frequency (lower = more responsive but more CPU), `redrawtime` limits syntax highlighting time (default 2000ms, increase for complex files), `synmaxcol` limits syntax highlighting column width (default 3000, reduce for files with very long lines). Your config sets `undolevels = 10000` which uses memory proportional to edit history. Swap files and shada (shared data -- the successor to viminfo) also affect performance: `shada` configuration controls how much command history, search patterns, and register content persists. For large files (>1MB), consider disabling some features with an autocommand.
**Key concepts:** updatetime, redrawtime, synmaxcol, undolevels, shada, swap files, large file handling, performance tuning
**Tip:** Create a "big file" autocommand that disables expensive features for files over 1MB: `vim.api.nvim_create_autocmd("BufReadPre", { callback = function() local ok, stats = pcall(vim.uv.fs_stat, vim.api.nvim_buf_get_name(0)) if ok and stats and stats.size > 1024*1024 then vim.b.bigfile = true; vim.opt_local.syntax = "off"; vim.treesitter.stop() end end })`. This prevents Neovim from freezing on large log files or data files.
**Tool anchor:** `:set updatetime?` to check current value; `:set redrawtime?`; `:syntime on`/`:syntime report` for syntax timing; `:checkhealth` for general diagnostics
**Drill:** Open a very large file (>10K lines) and observe how responsive scrolling and editing feels. Check `:set synmaxcol?` and try reducing it to 500 to see if scrolling improves. ### Config hot-reload
Iterating on your Neovim configuration requires reloading changes without restarting. `:source %` re-executes the current Lua file, but module caching means `require("custom.plugins.lspconfig")` returns the cached version unless you clear the cache first with `package.loaded["custom.plugins.lspconfig"] = nil`. Hot-reloading is imperfect -- some state (LSP servers, treesitter parsers, terminal sessions) cannot be cleanly reloaded.
**Key concepts:** :source, module cache, package.loaded, augroup clear, hot-reload limitations, <Leader>xx Lua eval
**Tip:** For rapid config iteration, use `<Leader>xx` on individual lines to test Lua expressions. For plugin spec changes (adding/removing plugins), you must run `:Lazy sync` since lazy.nvim manages the spec separately. Accept that some changes (LSP config, treesitter parsers) require a full Neovim restart.
**Tool anchor:** `<Leader>xx` for line/selection Lua eval; `:source %` for file reload; `:lua package.loaded["module"] = nil` to clear cache; `<Leader>L` for `:Lazy`

### Plugin update management
Lazy.nvim tracks exact plugin versions in `lazy-lock.json`, which pins every plugin to a specific git commit. Running `:Lazy update` fetches new versions and updates the lockfile.
