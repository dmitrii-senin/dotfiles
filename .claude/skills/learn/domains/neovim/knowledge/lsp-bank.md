# LSP, Completion, Diagnostics & Treesitter Topic Bank
Updated: 2026-05-31

## beginner

### What is LSP -- client-server model and lifecycle
The Language Server Protocol defines a JSON-RPC interface between an editor (client) and a language-specific server, decoupling IDE features from editor implementation. When Neovim attaches an LSP client to a buffer, it negotiates capabilities (what the server supports: completion, hover, diagnostics, formatting, etc.) through an initialize/initialized handshake. The server then receives document sync events (didOpen, didChange, didSave) and responds to requests (textDocument/definition, textDocument/completion). Understanding this client-server split explains why some features depend on server support and why different servers expose different capabilities for the same language.
**Key concepts:** JSON-RPC, capabilities negotiation, initialize/initialized lifecycle, textDocument sync, client attachment, server process
**Tip:** Run `:lua vim.print(vim.lsp.get_clients()[1].server_capabilities)` to see exactly what your clangd or pyright instance supports. If a feature like inlay hints does not work, the capability will be missing from this table and the problem is server-side, not your config.
**Tool anchor:** `:LspInfo` (or `:lua vim.print(vim.lsp.get_clients())` on 0.11+)
**Drill:** Open a C++ file and a Python file side by side. Run `:lua vim.print(vim.tbl_keys(vim.lsp.get_clients({bufnr=0})[1].server_capabilities))` in each. Compare the capability lists between clangd and pyright. Identify three capabilities that clangd supports but pyright does not (or vice versa) and explain what features those capabilities enable.
**Tags:** lsp, lifecycle, capabilities, json-rpc, client-server

### vim.lsp.config() and vim.lsp.enable() -- 0.11+ native setup
Neovim 0.11 introduced `vim.lsp.config(name, settings)` and `vim.lsp.enable(names)` as the native API for LSP server configuration, replacing the third-party nvim-lspconfig's `lspconfig[name].setup()` pattern. `vim.lsp.config()` registers a server configuration (cmd, settings, root_dir patterns, capabilities) under a name, while `vim.lsp.enable()` activates those named servers so they automatically attach when a matching filetype buffer opens. Your config uses `vim.lsp.config("clangd", { cmd = { "clangd", "--background-index", ... } })` then `vim.lsp.enable({ "lua_ls", "clangd", "rust_analyzer", "pyright" })` to bring all four servers online. The nvim-lspconfig plugin still provides the default configurations that `vim.lsp.config()` merges with.
**Key concepts:** vim.lsp.config, vim.lsp.enable, server name resolution, config merging, filetype matching, nvim-lspconfig defaults
**Tip:** You can split server configs across multiple files and call `vim.lsp.config()` additively -- the settings tables are deep-merged. This means you could put clangd's base config in your lspconfig.lua and override `cmd` args in an after/ftplugin/cpp.lua for project-specific flags like `--compile-commands-dir`.
**Tool anchor:** `:lua vim.print(vim.lsp.config.clangd)` to inspect the merged configuration
**Drill:** Add a new server to your config: configure `vim.lsp.config("bashls", { settings = {} })` and add `"bashls"` to the `vim.lsp.enable()` list. Open a .sh file, verify the server attaches with `:lua vim.print(vim.lsp.get_clients())`, then remove the config. Confirm you understand the relationship between the config name, the filetype, and the server binary.
**Tags:** lsp, vim-lsp-config, vim-lsp-enable, 0.11, setup, native-api

### Go-to-definition and references -- navigating code with LSP
Go-to-definition (`gd`) sends a `textDocument/definition` request to the server and jumps to where a symbol is defined; go-to-references (`grr`) sends `textDocument/references` and lists every usage. Your config overrides the built-in `gd` and `grr` to use Telescope pickers (`require("telescope.builtin").lsp_definitions` and `lsp_references`), which gives fuzzy filtering when multiple results exist. After jumping, `<C-o>` returns to the previous position via the jump list. For C++ with clangd, `gd` on a header declaration jumps to the implementation, while `gD` sends `textDocument/declaration` (useful for jumping to the header declaration from an implementation).
**Key concepts:** textDocument/definition, textDocument/references, textDocument/declaration, jump list, Telescope picker, gd, grr, gD, C-o
**Tip:** When `gd` returns multiple definitions (common with C++ overloaded functions or template specializations), the Telescope picker lets you preview each candidate before committing. Press `<C-s>` in your Telescope config to open the selected result in a horizontal split instead of replacing your current buffer.
**Tool anchor:** `gd` on any symbol, then `<C-o>` to return
**Drill:** In a C++ file, place your cursor on a class name and press `gd` to jump to its definition. Then place your cursor on a method call and press `grr` to see all references. Use the Telescope picker to filter results by filename. Return to your starting position using `<C-o>` repeatedly. Count how many jump list entries were created with `:jumps`.
**Tags:** lsp, go-to-definition, references, gd, grr, telescope, jump-list, navigation

### Completion -- native vim.lsp.completion
Neovim 0.11+ ships a built-in LSP completion source activated by `vim.lsp.completion.enable(true, client_id, bufnr, { autotrigger = true })`. Your config calls this in the LspAttach callback, so completion triggers automatically when you type a trigger character (`.` for member access, `::` for C++ scope, `(` for signature help). The popup menu (pum) shows candidates, `<C-n>`/`<C-p>` navigate, and your config maps `<Tab>` to confirm selection (`<C-y>`) when the pum is visible. The `completeopt = "menuone,noinsert,popup"` setting ensures a single match still shows the menu, the first item is not auto-inserted, and documentation appears in a popup window.
**Key concepts:** vim.lsp.completion.enable, autotrigger, completeopt, C-n/C-p navigation, Tab confirm, trigger characters, popup documentation
**Tip:** If completion feels slow on large C++ projects, the bottleneck is usually clangd indexing, not Neovim. Run `:lua vim.print(vim.lsp.get_clients({name="clangd"})[1].server_capabilities.completionProvider)` to check the server's trigger characters and resolve support. Adding `--completion-style=detailed` (which your config already has) gives richer completion items at the cost of slightly larger responses.
**Tool anchor:** `<C-n>` in insert mode to manually trigger, `<Tab>` to confirm
**Drill:** Open a C++ file, type `std::` and observe the autotrigger completion. Use `<C-n>` to navigate the list and `<Tab>` to confirm. Then press `<C-n>` on an empty line (where there is no trigger character) to manually invoke completion via your `<C-x><C-o>` mapping. Compare what appears in each case and note the difference between autotrigger and manual trigger.
**Tags:** lsp, completion, vim-lsp-completion, autotrigger, completeopt, popup-menu

### Diagnostics basics -- errors, warnings, signs, and navigation
Diagnostics are error/warning/info/hint messages that the LSP server (or external linters) push to Neovim asynchronously. Your config disables virtual_text in favor of `virtual_lines = { current_line = true }`, which renders the full diagnostic message as a virtual line below the current line only, avoiding visual clutter. Diagnostics have four severity levels (ERROR, WARN, INFO, HINT) displayed with Nerd Font icons in the sign column. You navigate errors with `]e`/`[e` (next/prev error) and warnings with `]w`/`[w`, both using `vim.diagnostic.jump()` filtered by severity. The `<Leader>dd` mapping opens Trouble for buffer diagnostics, and `<Leader>dw` for workspace diagnostics.
**Key concepts:** diagnostic severity (ERROR, WARN, INFO, HINT), virtual_text, virtual_lines, sign column, ]e/[e navigation, ]w/[w, diagnostic float, Trouble
**Tip:** The `current_line = true` setting for virtual_lines is the sweet spot: you see the full diagnostic text only for the line you are on, avoiding the screen-space cost of showing all virtual lines simultaneously. If you temporarily need all virtual lines visible (during a refactor with many errors), toggle with `:lua vim.diagnostic.config({ virtual_lines = true })`.
**Tool anchor:** `]e` to jump to next error, `<Leader>dd` for Trouble buffer diagnostics
**Drill:** Introduce a deliberate type error in a C++ file (e.g., assign a string to an int). Watch the diagnostic appear in the sign column. Move your cursor to the error line and observe the virtual line rendering. Press `<Leader>df` to see the diagnostic float. Press `<Leader>dd` to open Trouble. Fix the error and confirm the diagnostic disappears. Then press `<Leader>dt` to toggle diagnostics off and on.
**Tags:** lsp, diagnostics, virtual-lines, severity, signs, navigation, trouble

### Hover and signature help -- K and floating windows
Pressing `K` in normal mode sends a `textDocument/hover` request and displays the result in a floating window -- typically the type signature and documentation for the symbol under the cursor. For C++ with clangd, hover shows the full declaration, including template parameters, return type, and Doxygen comments. Signature help (triggered automatically when you type `(` in insert mode) shows the function signature with the current parameter highlighted. Both use Neovim's floating window infrastructure with configurable border, size, and focus behavior. Pressing `K` again while a hover float is open focuses the float, allowing you to scroll through long documentation with `<C-d>`/`<C-u>`.
**Key concepts:** textDocument/hover, textDocument/signatureHelp, floating window, K mapping, parameter highlighting, Doxygen rendering
**Tip:** Hover is one of the most underused LSP features. In C++ codebases with SBE-generated types, hovering over a message field instantly shows you the type and offset without opening the schema file. Get in the habit of pressing K before reaching for grep.
**Tool anchor:** `K` on any symbol to hover, type `(` after a function name in insert mode for signature help
**Drill:** Open a C++ file with template code. Place your cursor on a template class name and press `K` to see the hover info. Then press `K` again to focus the float and scroll with `<C-d>`. Close with `q`. Move to a function call, enter insert mode, delete and retype the `(` to trigger signature help. Observe the parameter highlighting as you type each argument.
**Tags:** lsp, hover, signature-help, floating-window, K, documentation

### Code actions -- quick fixes, refactors, and auto-import
Code actions are context-sensitive operations offered by the LSP server for the current cursor position or selection. Your config maps `<Leader>ca` to `vim.lsp.buf.code_action()`, which pops up a menu of available actions. Common code actions from clangd include: add missing `#include`, extract variable/function, apply clang-tidy fix-it, convert raw pointer to smart pointer, and add missing `override` specifier. Actions are categorized as quickfix (fixes a diagnostic), refactor (structural code change), or source (file-wide operations like organize imports). Some code actions can be applied to a visual selection, enabling "extract function" workflows.
**Key concepts:** textDocument/codeAction, quickfix, refactor, source actions, auto-import, clang-tidy fix-its, selection-based actions
**Tip:** When clangd reports a missing include via diagnostics, pressing `<Leader>ca` on the error line usually offers "add #include" as the first code action. This is faster than manually finding and typing the header path, especially for unfamiliar APIs in SBE or CME MDP 3.0 codebases.
**Tool anchor:** `<Leader>ca` on a diagnostic line or visual selection
**Drill:** Write a C++ function that uses `std::vector` without including the header. Wait for the diagnostic, then press `<Leader>ca` and select the "add #include" action. Next, select a block of code in visual mode and press `<Leader>ca` to see refactoring options. Try extracting the selection to a function if clangd offers it.
**Tags:** lsp, code-actions, quickfix, refactor, auto-import, clang-tidy

### Treesitter basics -- parsing, highlighting, and vim.treesitter.start()
Tree-sitter is an incremental parser that builds a concrete syntax tree (CST) for each buffer, enabling accurate syntax highlighting, code folding, text objects, and structural navigation. Your config installs parsers for 30+ languages (including cpp, c, rust, python, lua, asm, disassembly) via `require("nvim-treesitter").install(parsers)` and activates highlighting with a FileType autocmd that calls `pcall(vim.treesitter.start)`. Unlike regex-based highlighting, tree-sitter understands nesting (a string inside a function inside a class), so highlighting is always correct even with complex C++ template syntax. Tree-sitter and LSP serve complementary roles: tree-sitter handles local syntax structure while LSP provides semantic understanding.
**Key concepts:** incremental parsing, syntax tree, parser installation, vim.treesitter.start, FileType autocmd, concrete syntax tree, language grammars
**Tip:** If highlighting looks wrong for a specific file, run `:InspectTree` to see the parsed syntax tree. If the tree shows ERROR nodes, the parser may need updating (`:TSUpdate`) or the file may have genuinely malformed syntax. For C++ files with heavy macro usage, tree-sitter may show errors in macro bodies since it parses the pre-preprocessor source.
**Tool anchor:** `:InspectTree` to view the syntax tree, `:TSUpdate` to update parsers
**Drill:** Open a C++ file and run `:InspectTree` to see the syntax tree in a split. Click on different nodes in the tree and observe which source code region highlights. Then open a Lua file and compare the tree structure. Close the tree with `:q`. Try temporarily disabling tree-sitter highlighting with `:lua vim.treesitter.stop()` and notice how the highlighting changes to regex-based fallback. Re-enable with `:lua vim.treesitter.start()`.
**Tags:** treesitter, parsing, highlighting, syntax-tree, parsers, vim-treesitter-start

### Mason.nvim -- installing and managing LSP servers
Mason.nvim is a portable package manager that installs LSP servers, formatters, linters, and DAP adapters into a Neovim-managed directory (`~/.local/share/nvim/mason/`). Your config lists mason.nvim as a dependency of nvim-lspconfig, ensuring Mason's binary directory is on PATH before servers are started. Your health.lua verifies Mason-managed binaries: clangd, rust-analyzer, pyright, lua-language-server, stylua, clang-format, ruff, codelldb, and debugpy. Mason handles platform-specific installation (downloading prebuilt binaries or building from source), version pinning, and updates through the `:Mason` interactive UI. The separation between Mason (installation) and vim.lsp.config/enable (configuration) means you can also install servers outside Mason and they work as long as the binary is on PATH.
**Key concepts:** Mason, package management, binary installation, :Mason UI, :MasonToolsInstall, mason bin directory, PATH, health check verification
**Tip:** When `:checkhealth custom` reports a missing Mason binary, `:Mason` opens the interactive installer where you can search and install any supported tool. For CI or headless setup, `:MasonToolsInstall` (from mason-tool-installer.nvim if configured) batch-installs all specified tools. Your health check distinguishes between Mason-installed and PATH-installed binaries, accepting both.
**Tool anchor:** `:Mason` to open the installer UI, `:checkhealth custom` to verify all binaries
**Drill:** Open `:Mason` and browse the available packages. Install a new tool you have not used before (like `cpplint` or `cppcheck`). Verify it appears in the mason bin directory: `:lua vim.print(vim.fn.stdpath("data") .. "/mason/bin/")`. Then uninstall it from the Mason UI. Run `:checkhealth custom` and compare the output before and after installing a tool that your health check monitors.
**Tags:** mason, package-management, lsp-servers, formatters, linters, installation, health-check

### Snippet jumping -- C-l, C-h, and vim.snippet
Neovim 0.10+ ships a built-in snippet engine (`vim.snippet`) that expands LSP snippet completions (like function signatures with placeholders) and supports tab-stop navigation. Your config maps `<C-l>` and `<C-h>` in insert and select modes to `vim.snippet.jump(1)` and `vim.snippet.jump(-1)` respectively, but only when a snippet is active (checked with `vim.snippet.active({ direction = 1 })`). When you accept a completion that includes a snippet (like a function with parameters), the cursor enters the first placeholder, `<C-l>` jumps to the next placeholder, and `<C-h>` goes back. In normal mode, `<C-l>` and `<C-h>` retain their window navigation behavior. This built-in snippet support replaces the need for external snippet engines like LuaSnip for LSP-provided snippets.
**Key concepts:** vim.snippet, snippet.jump, snippet.active, tab stops, placeholders, insert mode, select mode, LSP snippet completion, C-l/C-h
**Tip:** When clangd provides a function completion with `--function-arg-placeholders` (enabled in your config), accepting the completion inserts the function with placeholder arguments. Press `<C-l>` to jump between arguments, typing the value for each. If you accidentally exit the snippet, the placeholders remain as text that you can edit manually.
**Tool anchor:** Accept a snippet completion (function with parameters), then `<C-l>` to jump forward, `<C-h>` to jump backward
**Drill:** Open a C++ file and type a function name that has parameters (like `std::sort(`). Accept the completion that includes placeholders. Observe the first placeholder is selected. Press `<C-l>` to jump to the next parameter placeholder, type a value, then `<C-l>` again. Try `<C-h>` to jump backward. Then check if a snippet is active: type in insert mode and verify `<C-l>` does nothing when no snippet is active (falls through to default behavior).
**Tags:** snippets, vim-snippet, jump, placeholders, completion, C-l, C-h, insert-mode

### Diagnostic configuration -- virtual_text, virtual_lines, severity_sort
The `vim.diagnostic.config()` function controls how diagnostics render across all sources. Your config uses `virtual_text = false` and `virtual_lines = { current_line = true }` with Nerd Font icons for sign text. Key options include: `severity_sort = true` (higher-severity diagnostics render on top), `update_in_insert = false` (default, delays updates until leaving insert mode to avoid flicker), `float = { border = "rounded", source = true }` (shows source in floating diagnostics), and `underline = true` (underlines diagnostic ranges). The `virtual_lines` option uses the lsp_lines rendering strategy where the full diagnostic message appears as a virtual line below the source line, with a connecting line drawn to the exact error position.
**Key concepts:** vim.diagnostic.config, virtual_text, virtual_lines, current_line, severity_sort, update_in_insert, signs, underline, float
**Tip:** For focused debugging sessions where you want to see all diagnostics at once, temporarily switch to full virtual_lines: `:lua vim.diagnostic.config({ virtual_lines = true })`. When the noise becomes too much, switch back: `:lua vim.diagnostic.config({ virtual_lines = { current_line = true } })`. Consider binding this toggle to a keymap for rapid switching during refactoring.
**Tool anchor:** `:lua vim.diagnostic.config({ virtual_lines = { current_line = true } })` to restore your default
**Drill:** Open a file with several diagnostics. Run `:lua vim.diagnostic.config({ virtual_text = true, virtual_lines = false })` and compare the rendering. Then try `{ virtual_lines = true }` and `{ virtual_lines = { current_line = true } }` side by side. Finally, try `{ severity_sort = true }` vs `{ severity_sort = false }` on a line with multiple diagnostics of different severities. Document which configuration you prefer for different workflows.
**Tags:** diagnostics, virtual-lines, virtual-text, configuration, severity-sort, rendering

### Inlay hints -- vim.lsp.inlay_hint.enable and toggle
Inlay hints are inline annotations rendered by the LSP server showing inferred types, parameter names, and other contextual information that is not in the source code. For C++ with clangd, inlay hints show deduced `auto` types, parameter names at call sites (`/*size=*/`), and return types for lambdas. Your config maps `<Leader>th` to toggle hints per-buffer using `vim.lsp.inlay_hint.enable(not vim.lsp.inlay_hint.is_enabled({ bufnr = event.buf }))`, and this mapping only appears if the attached server supports `textDocument/inlayHint`. Hints are rendered using extmarks with a special highlight group (LspInlayHint) and can be styled via `vim.api.nvim_set_hl`.
**Key concepts:** vim.lsp.inlay_hint.enable, textDocument/inlayHint, extmarks, auto type deduction, parameter names, LspInlayHint highlight, capability check
**Tip:** Inlay hints are invaluable when reading unfamiliar C++ code heavy with `auto` and template type deduction, but they add visual noise in code you wrote yourself. Toggle them on with `<Leader>th` when reading, off when writing. For SBE-generated code where every type is explicit, hints add no value and should stay off.
**Tool anchor:** `<Leader>th` to toggle inlay hints
**Drill:** Open a C++ file with `auto` variables and lambda expressions. Press `<Leader>th` to enable inlay hints. Observe the deduced types appearing inline. Then find a function call with multiple arguments and note the parameter name hints. Press `<Leader>th` again to disable. Customize the hint appearance with `:lua vim.api.nvim_set_hl(0, "LspInlayHint", { fg = "#6c7086", italic = true })`.
**Tags:** lsp, inlay-hints, toggle, auto-type, parameter-names, extmarks

### Document symbols and outline -- gO and code navigation
Document symbols (`textDocument/documentSymbol`) provide a structured outline of all symbols in the current buffer: functions, classes, methods, variables, types, organized hierarchically. Your config maps `gO` to `require("telescope.builtin").lsp_document_symbols`, which presents this hierarchy in a Telescope picker for fuzzy searching. This is essential for navigating large C++ files: instead of scrolling through a 2000-line file, type `gO` then fuzzy-match the function name. The hierarchy matters -- a method inside a class appears nested, so you can filter by class name to narrow results. This replaces the need for ctags-based tagbar plugins.
**Key concepts:** textDocument/documentSymbol, symbol hierarchy, Telescope lsp_document_symbols, gO mapping, SymbolKind (function, class, method, variable)
**Tip:** In large C++ files with many overloaded methods, the Telescope symbol picker lets you type partial signatures to disambiguate. For example, typing `decode msg` would match `void decode(const MsgHeader&)` even with many other `decode` overloads.
**Tool anchor:** `gO` to open the document symbol picker
**Drill:** Open a C++ file with multiple classes and functions. Press `gO` and observe the symbol hierarchy. Type a partial function name to filter. Press Enter to jump to a symbol. Then try `:Telescope lsp_document_symbols symbols=function` to filter by symbol kind. Compare the speed of `gO` + fuzzy search vs manual `/` search for finding a specific function in a large file.
**Tags:** lsp, document-symbols, outline, gO, telescope, navigation, symbol-kind

### Workspace symbols -- gW and project-wide search
Workspace symbols (`textDocument/workspaceSymbol`) search across your entire project, not just the current file. Your config maps `gW` to `require("telescope.builtin").lsp_dynamic_workspace_symbols`, which queries the server as you type (dynamic means the query is sent on each keystroke, not preloaded). For clangd, this searches the background index built from your compile_commands.json, covering every translation unit. This is how you jump to a symbol definition in a different file without knowing the filename: type `gW`, enter the class or function name, and jump directly. This is significantly more precise than grep because it understands scope, overloads, and namespaces.
**Key concepts:** workspace/symbol, dynamic query, background index, cross-file navigation, gW mapping, compile_commands.json, symbol search
**Tip:** Workspace symbol search with clangd respects C++ namespaces. Searching for `OrderBook::insert` will match only the method, not unrelated `insert` functions. For SBE codecs with generated namespaces, searching the full qualified name (e.g., `mktdata::MDIncrementalRefreshBook46`) instantly locates the generated class.
**Tool anchor:** `gW` to open workspace symbol search
**Drill:** With clangd running on a multi-file project, press `gW` and type a class name that you know exists in a different file. Jump to it. Then search for a common function name like `init` or `process` and observe how many matches appear across the project. Use the Telescope previewer to inspect each match before jumping. Compare this workflow to `<Leader>fs` (live grep) for finding the same symbol.
**Tags:** lsp, workspace-symbols, gW, cross-file, dynamic-search, telescope, clangd-index

### Treesitter text objects -- select, move, swap
The nvim-treesitter-textobjects plugin provides structural text objects based on the syntax tree rather than regex patterns. Your config defines: `af`/`if` for outer/inner function, `ac`/`ic` for outer/inner class, `aa`/`ia` for outer/inner parameter. Movement keys `]m`/`[m` jump to the next/previous function start, `]]`/`[[` jump to class boundaries, and `]M`/`[M` to function ends. The swap keys `<Leader>a`/`<Leader>A` swap the current parameter with the next/previous one. These text objects use tree-sitter queries (like `@function.outer`) that match syntactic nodes, so `daf` deletes an entire C++ function including its signature and body, regardless of how many lines it spans. The `lookahead = true` setting means pressing `af` when not inside a function selects the next function forward.
**Key concepts:** @function.outer/inner, @class.outer/inner, @parameter.outer/inner, ]m/[m movement, ]]/ [[ class movement, <Leader>a swap, lookahead, set_jumps
**Tip:** The parameter swap (`<Leader>a`/`<Leader>A`) is a huge time-saver when refactoring function signatures. In C++ code with many parameters (common in SBE message constructors), you can reorder arguments structurally without cut-paste errors. Combine with `daa` (delete a parameter including comma) and `p` to move parameters around.
**Tool anchor:** `vaf` to visually select outer function, `]m` to jump to next function
**Drill:** Open a C++ file with several functions and classes. Use `vaf` to select a function, then `vac` to select a class. Navigate with `]m` (next function start), `[m` (previous), `]]` (next class). On a function with multiple parameters, place cursor on a parameter and press `<Leader>a` to swap it with the next one. Try `daa` to delete a parameter (including its comma), then `p` to paste it elsewhere.
**Tags:** treesitter, text-objects, select, move, swap, function, class, parameter

### Treesitter and LSP folding -- dual-source foldexpr
Your config sets up a two-layer folding strategy: the base is `vim.treesitter.foldexpr()` (set globally in treesitter.lua), which creates folds based on syntax tree nodes (functions, classes, blocks). When an LSP client attaches that supports `textDocument/foldingRange`, your LspAttach callback overrides the buffer's foldexpr to `vim.lsp.foldexpr()`, which uses the server's semantic understanding for folds. LSP folding can be smarter than tree-sitter folding because it understands constructs like C++ `#pragma region` or Python docstrings that are not purely syntactic. The `foldlevel = 99` setting keeps all folds open by default so they do not obstruct reading.
**Key concepts:** foldmethod=expr, vim.treesitter.foldexpr, vim.lsp.foldexpr, foldlevel, window-local options, dual-source strategy, za toggle, zM/zR
**Tip:** The `vim.wo[win][0].foldmethod` syntax sets a window-local, buffer-local option (the `[0]` is the buffer scope). This means the LSP fold override only applies to the window where the buffer was opened, not globally. If you split the same buffer, the new window gets treesitter folds until its own LspAttach fires.
**Tool anchor:** `za` to toggle a fold, `zM` to close all, `zR` to open all, `zc`/`zo` to close/open one
**Drill:** Open a C++ file and press `zM` to close all folds. Observe the fold structure (functions, classes, namespaces). Press `zR` to reopen all. Use `za` on a specific function to toggle its fold. Then check which foldexpr is active: `:set foldexpr?`. Kill the LSP client with `:lua vim.lsp.stop_client(vim.lsp.get_clients()[1].id)` and check foldexpr again to see if it reverts to treesitter. Restart with `:edit`.
**Tags:** treesitter, lsp, folding, foldexpr, foldlevel, dual-source, window-local

### Formatting pipeline -- conform.nvim with format-on-save
Your config uses conform.nvim as the formatting layer, decoupled from LSP. It maps specific formatters to filetypes: clang_format (Google style) for C/C++, ruff_format for Python, rustfmt for Rust, stylua for Lua. `format_on_save` runs the formatter automatically on every write with a 1-second timeout and LSP fallback (if no conform formatter is configured, it uses the LSP server's formatting). The `<Leader>cf` mapping triggers manual formatting for both normal mode (whole file) and visual mode (range). This separation from LSP formatting means you can use a different formatter than what the LSP server provides, which is important when your team enforces specific formatter versions or configs.
**Key concepts:** conform.nvim, formatters_by_ft, format_on_save, lsp_fallback, timeout_ms, <Leader>cf manual format, clang_format, stylua, rustfmt
**Tip:** The `--style=Google` flag on clang_format in your config applies Google's C++ style guide. For projects with a `.clang-format` file in the root, clang_format will use that file instead, and the `--style=Google` becomes a fallback. Check which style is actually used with `:!clang-format --dump-config` in the project root.
**Tool anchor:** `<Leader>cf` to format manually, `:w` triggers format-on-save

### Call hierarchy -- incoming and outgoing calls

### Performance tuning -- debounce, incremental sync, large projects
LSP performance degrades on large C++ projects through several bottlenecks: document sync (sending full file content on every keystroke), diagnostics computation (re-analyzing after every change), file watching (tracking thousands of header files), and completion latency. Key tuning options: set `flags.debounce_text_changes` to batch didChange notifications (300-500ms reduces traffic without noticeable latency), use incremental sync (`TextDocumentSyncKind.Incremental = 2`) to send only changed regions instead of full file content, and configure clangd's `--limit-results=50` to cap completion candidates. For background indexing, clangd's `--background-index-priority=low` prevents the indexer from competing with interactive requests. Memory usage scales with project size: clangd can consume 4-8GB for large monorepos.
**Key concepts:** debounce_text_changes, TextDocumentSyncKind, incremental vs full sync, file watching, --limit-results, background-index-priority, memory usage
**Tip:** If clangd is slow on your market data C++ project, check if it is using incremental sync: `:lua vim.print(vim.lsp.get_clients({name="clangd"})[1].server_capabilities.textDocumentSync)`. Value 2 means incremental (good), value 1 means full (every keystroke sends the entire file). If full, upgrade clangd. Also check `.clangd` for `Background: Skip` to disable background indexing of test files.
**Tool anchor:** `:lua vim.print(vim.lsp.get_clients({name="clangd"})[1].server_capabilities.textDocumentSync)` to check sync mode
**Drill:** Measure your LSP responsiveness: open a large C++ file (1000+ lines), time how long completion takes after typing `.`, and time go-to-definition.
