# Navigation Topic Bank
Updated: 2026-05-31

## beginner

### Buffers
A buffer is Neovim's in-memory representation of a file. `:e filename` opens a file into a new buffer, `:b name` switches to a buffer by partial name match, `:bd` deletes (unloads) the current buffer, and `:ls` lists all buffers with their status flags. A buffer can be loaded but not displayed in any window (hidden), and your config sets `autowrite = true`, which automatically saves modified buffers when you switch away. Understanding that buffers are independent of windows is fundamental -- one buffer can appear in multiple windows, and switching buffers does not close files.
**Key concepts:** buffer, :e, :b, :bd, :ls, hidden buffers, buffer vs file, loaded/listed/hidden states, buffer lifecycle
**Tip:** Your `autowrite = true` setting means you rarely need to manually `:w` before switching buffers -- Neovim saves automatically. This makes buffer-switching with `:b` or `<Leader>fb` (Telescope buffers) frictionless. But be aware: `autowrite` only triggers on certain actions (`:next`, `:b`, window switches), not on every possible leave event. If you want to be safe, `<C-s>` (your save mapping) before any risky operation.
**Tool anchor:** `:help buffers`, `:help :ls`, `:help 'autowrite'`
**Drill:** Open three C++ files from your SBE project with `:e` commands. Run `:ls` and identify the buffer numbers and flags (`%a` for active, `#` for alternate, `h` for hidden). Switch between them using `:b partial_name` with Tab completion. Delete one buffer with `:bd` and verify it disappears from `:ls`. Open it again and confirm it gets a new buffer number. Practice using `<Leader>fb` (Telescope buffers) to fuzzy-search through your open buffers.
**Tags:** buffers, buffer-list, hidden, autowrite, buffer-lifecycle

### Window splits
Windows are viewports into buffers. `:sp` (or `C-w s`) creates a horizontal split, `:vsp` (or `C-w v`) creates a vertical split. Navigate between windows with `C-h/j/k/l` (your zellij-nav remappings, which also cross into Zellij terminal panes). Rearrange windows with `C-w H/J/K/L` to push a window to the far left/bottom/top/right of the screen. `C-w o` closes all windows except the current one. Each window independently controls cursor position, folds, and scroll state for its buffer.
**Key concepts:** :sp, :vsp, C-w s/v, C-w h/j/k/l, C-w H/J/K/L rearrange, C-w o, window vs buffer
**Tip:** Your zellij-nav.nvim plugin remaps `C-h/j/k/l` to navigate seamlessly between Neovim splits and Zellij terminal panes. This means the same keys work whether you are moving between two Neovim windows or from Neovim to an adjacent terminal running your C++ build. For Neovim-only window management, your `<Leader>ws` and `<Leader>wv` mappings in keymaps.lua are equivalent to `C-w s` and `C-w v`.
**Tool anchor:** `:help windows`, `:help CTRL-W`, `:help :split`
**Drill:** Open a C++ header and source file side by side: `:e header.h`, then `:vsp source.cpp`. Navigate between them with `C-h` and `C-l`. Now open a Zellij terminal pane below and verify `C-j` moves from Neovim into the terminal pane. Back in Neovim, use `C-w L` to push the right window to the far right, then `C-w H` to push it left. Use `<Leader>wd` to close a window (not the buffer). Confirm the buffer is still in `:ls`.
**Tags:** windows, splits, zellij-nav, window-navigation, C-w

### Window sizing
`C-w =` equalizes all window sizes. `C-w |` maximizes width, `C-w _` maximizes height, and `C-w +/-` adjusts height by one line. Your config maps `<M-S-Arrow>` keys for resizing: Alt-Shift-Up/Down adjusts height by 2 lines, Alt-Shift-Left/Right adjusts width by 2 columns. The zoom toggle `<Leader>wz` maximizes the current window (both width and height) and restores the previous layout when pressed again, using a custom function that tracks the zoomed state in `vim.t.zoom_winid`.
**Key concepts:** C-w =, C-w |, C-w _, C-w +/-, zoom toggle, resize mappings, equalize
**Tip:** Your `<Leader>wz` zoom toggle is implemented as a custom function in keymaps.lua that calls `wincmd |` and `wincmd _` to maximize, and `wincmd =` to restore. This is ideal for temporarily focusing on one file in a multi-split layout: zoom in to read or edit, then zoom out to see context. The state is tracked per tab page via `vim.t.zoom_winid`, so each tab has independent zoom state.
**Tool anchor:** `:help CTRL-W_=`, `:help CTRL-W_|`, `:help window-resize`
**Drill:** Create a three-way split layout: one main file and two reference files. (1) Use `C-w =` to equalize all panes. (2) Focus on the main file and use `<Leader>wz` to zoom it full-screen. Edit some code. (3) Press `<Leader>wz` again to restore the three-pane layout. (4) Use `<M-S-Right>` a few times to widen the current pane incrementally. (5) Reset with `C-w =`. Practice the zoom-edit-restore cycle until it becomes your default workflow for focused editing.
**Tags:** window-sizing, zoom, equalize, resize, leader-wz

### Tab pages
Tab pages are collections of windows. `:tabnew` (or `<Leader><tab><tab>` in your config) creates a new tab, `gt`/`gT` or `]<tab>`/`[<tab>` navigate between tabs, and `:tabclose` (or `<Leader><tab>d`) closes the current tab. Tabs are best used for separate contexts or tasks (e.g., one tab for the SBE codec, another for the order book, a third for tests) rather than as a substitute for buffers. Buffers are shared across tabs, so `:b name` works regardless of which tab you are in.
**Key concepts:** :tabnew, gt/gT, :tabclose, tabs as context containers, buffers shared across tabs, tab vs buffer
**Tip:** Your keymaps.lua provides a full tab workflow: `<Leader><tab><tab>` for new tab, `<Leader><tab>d` to close, `<Leader><tab>l`/`<Leader><tab>f` for last/first tab, `<Leader><tab>o` to close all other tabs. Use tabs sparingly -- one for each logical task (coding, debugging, reference) -- and use buffers within each tab. Too many tabs defeats the purpose; if you have more than 4, you likely need a project/session manager instead.
**Tool anchor:** `:help tabpage`, `:help gt`, `:help :tabnew`
**Drill:** Create three tabs: (1) `<Leader><tab><tab>` for a coding tab, open your main C++ source; (2) `<Leader><tab><tab>` for a test tab, open the corresponding test file; (3) `<Leader><tab><tab>` for a reference tab, open a header file. Navigate between them with `gt`/`gT` and `]<tab>`/`[<tab>`. Use `<Leader><tab>o` to close all but the current tab. Rebuild the layout. Note that `:b source.cpp` works from any tab because buffers are global.
**Tags:** tab-pages, context-switching, tabnew, gt, tabclose

### Marks
`m{a-z}` sets a local mark (per-buffer) and `m{A-Z}` sets a global mark (across buffers). Jump to a mark's line with `'{x}` or to its exact position with `` `{x} ``. Special marks: `''` is the position before the last jump, `'.` is the position of the last change, `'[` and `']` bracket the last changed or yanked text, and `'<`/`'>` bracket the last visual selection. Local marks (a-z) are forgotten when you close the buffer, but global marks (A-Z) persist across sessions (saved in shada).
**Key concepts:** m{a-z} local, m{A-Z} global, '/`, special marks ('', '., '[, ']), mark persistence, shada
**Tip:** Set global marks on files you frequently jump to: `mS` on your SBE schema definition, `mH` on your main handler, `mT` on your test file. Then `'S` from anywhere instantly opens that file and jumps to the marked line. Global marks survive across sessions via shada, so you set them once and use them indefinitely. Use `<Leader>f'` (your Telescope marks picker) to browse all marks.
**Tool anchor:** `:help mark-motions`, `:help '`, `:help :marks`
**Drill:** In your project: (1) open three key files and set global marks `mA`, `mB`, `mC` on important lines in each; (2) from a completely different file, jump to each mark with `'A`, `'B`, `'C` and verify you land in the right file and line; (3) use special marks: make a change, then `'.` to jump back to it; jump somewhere far, then `''` to return; (4) browse all marks with `<Leader>f'` in Telescope. (5) Close and reopen Neovim -- verify global marks persist.
**Tags:** marks, global-marks, local-marks, special-marks, navigation-anchors

### Jump list and change list
The jump list records positions before each "jump" (searches, marks, `:tag`, `gd`, etc.). `C-o` goes back to the previous jump location, `C-i` goes forward. The change list records the position of every change: `g;` goes to the previous change, `g,` goes to the next. These are separate stacks -- the jump list tracks where you navigated, the change list tracks where you edited. Your config sets `jumpoptions = "view"`, which restores the scroll position (not just cursor) when navigating the jump list.
**Key concepts:** C-o/C-i jump list, g;/g, change list, jump vs change, jumpoptions=view, stack navigation
**Tip:** Your `jumpoptions = "view"` setting is a significant quality-of-life improvement: when you press `C-o` to go back, Neovim restores not just the cursor position but the exact scroll state (which line was at the top of the screen). This means jumping back truly returns you to what you were looking at, not just where the cursor was. The `view` flag has been available since Neovim 0.8, but most configs do not have it.
**Tool anchor:** `:help jumplist`, `:help changelist`, `:help 'jumpoptions'`
**Drill:** Open a C++ source file. Use `gd` to go to a definition, then `/` to search for something, then `*` to search for the word under cursor. Now trace back: (1) `C-o` three times to walk back through all three jumps; (2) `C-i` to go forward again. Now use `g;` to jump to your last change location (different from the last jump). Practice the mental model: "C-o goes where I was looking, g; goes where I was editing." Verify that `jumpoptions=view` restores scroll position with `C-o`.
**Tags:** jump-list, change-list, C-o, C-i, g-semicolon, jumpoptions

### File explorer with oil.nvim
Oil.nvim treats a directory as an editable buffer where filenames are text lines. To create a file, type a new filename on an empty line; to rename, edit the filename text; to delete, delete the line. Changes are staged and applied when you save the buffer (`:w`). Your config opens oil in a floating window with `<Leader>-` (parent directory of current file) and `<Leader>_` (cwd). Navigate with `-` to go up a directory and `<CR>` to enter a directory or open a file.
**Key concepts:** directory as buffer, create/rename/delete files, <Leader>-/<Leader>_, floating window, stage-on-save
**Tip:** Oil's "directory as buffer" model means you can use all your Neovim editing skills for file management: `yy` to copy a filename, `p` to create a duplicate entry, `dd` to stage a deletion, `:s/old/new/` to batch-rename files. Your config disables `C-h/j/k/l` in oil (to not conflict with zellij-nav) and uses `C-s`/`C-v`/`C-t` for opening files in horizontal/vertical splits or new tabs. Press `g.` to toggle hidden files.
**Tool anchor:** `:help oil`, `:help oil-actions`
**Drill:** Open oil with `<Leader>-` to see the parent directory of your current file. (1) Navigate into a subdirectory with `<CR>` and back up with `-`. (2) Create a new file: press `o` to open a new line, type `test_new.cpp`, and save with `:w` to create the file. (3) Rename a file by editing its name text and saving. (4) Delete a file by `dd` on its line and saving. (5) Open a file in a vertical split with `C-v`. (6) Toggle hidden files with `g.`. Practice the flow of navigating, creating, and renaming without leaving Neovim.
**Tags:** oil-nvim, file-explorer, directory-buffer, file-management, floating-window

### Telescope basics
Telescope is a fuzzy finder framework. Your config maps `<Leader>ff` for find files, `<Leader>fs` for live grep (search string across project), `<Leader>fb` for open buffers, and `<Leader>fh` for help tags. The Telescope UI has insert mode (for typing queries) and normal mode (for navigating results). `<CR>` opens in the current window, `C-s` opens in a horizontal split, `C-v` in a vertical split, `C-t` in a new tab. The fzf-native extension provides fast sorting, and `path_display = "smart"` shortens paths intelligently.
**Key concepts:** fuzzy finder, <Leader>ff/<Leader>fs/<Leader>fb/<Leader>fh, insert vs normal in picker, split actions, fzf-native
**Tip:** `<Leader>fs` (live grep) uses `rg` under the hood (since your `grepprg = "rg --vimgrep"`), so it respects `.gitignore` and is fast even on large codebases. When searching your SBE codec project, type a function name and results appear instantly. To narrow further, use Telescope's insert-mode filtering: the query is a fuzzy match, so typing `decmsg` matches `decodeMessage`, `decode_msg`, etc.
**Tool anchor:** `:help telescope.nvim`, `:help telescope.mappings`
**Drill:** (1) Use `<Leader>ff` to find a file by partial name in your project. (2) Use `<Leader>fs` to search for a function name across all files -- note how results update as you type. (3) Open a result in a vertical split with `C-v`. (4) Use `<Leader>fb` to switch between open buffers by fuzzy name. (5) Use `<Leader>fh` to search Neovim help for "operator" -- this is faster than `:help operator`. (6) Inside any Telescope picker, press `<Esc>` to enter normal mode, use `j/k` to navigate, and `<CR>` to select.
**Tags:** telescope, fuzzy-finder, find-files, live-grep, buffers, fzf

### LSP navigation basics
Language Server Protocol (LSP) provides semantic code navigation powered by your language server. Your config attaches Telescope pickers to LSP actions: `gd` jumps to definition, `gD` to declaration, `grr` shows all references, `gri` shows implementations, `gO` lists document symbols (functions, classes, variables in the current file), and `gW` searches workspace symbols across the project. These are semantic -- they understand C++ scopes, overloads, and namespaces, unlike grep-based text search. LSP navigation is the primary way to read and understand unfamiliar code.
**Key concepts:** gd, gD, grr, gri, gO, gW, definition vs declaration, references, document symbols, workspace symbols
**Tip:** `gO` (document symbols) is an underused power tool: it gives you an instant outline of the current file's structure -- all functions, classes, namespaces, and variables. In a large C++ file with your SBE decoders, `gO` shows every method at a glance, and selecting one jumps directly to it. It is faster than scrolling or searching when you know the symbol name approximately.
**Tool anchor:** `:help lsp`, `:help vim.lsp.buf.definition()`, `:help vim.lsp.buf.references()`
**Drill:** (1) Open a C++ source file with clangd attached. Use `gd` on a function call to jump to its definition. (2) From the definition, use `grr` to find all callers -- Telescope shows each reference with preview. (3) Use `gO` to see the document outline and jump to a specific method. (4) Use `gW` to search for a symbol across the entire workspace by name. (5) Use `gD` on a function to see its declaration (typically in the header). (6) Practice the cycle: `gd` to understand, `grr` to find usage, `gO` to orient yourself in a file.
**Tags:** LSP, definition, references, symbols, clangd, code-navigation

### Help navigation
Neovim's built-in help system is a full hypertext documentation browser. `:help topic` opens help, `C-]` follows a tag (link) under the cursor, `C-t` jumps back in the tag stack. Help topics use naming conventions: `'option'` for options, `:command` for ex commands, `key` for normal-mode keys, `i_key` for insert-mode keys, `v_key` for visual-mode keys. Your `<Leader>fh` Telescope picker fuzzy-searches all help tags, and `<Leader>hh` looks up help for the word under the cursor. Learning to navigate help efficiently is a force multiplier for all other Neovim skills.
**Key concepts:** :help, C-] follow tag, C-t tag stack, help tag naming, <Leader>fh, <Leader>hh, help as hypertext
**Tip:** Your keymaps.lua maps `<Leader>hh` to look up help for the word under cursor and `<Leader>hH` for the WORD under cursor. When reading Neovim source or config files and you encounter an unfamiliar function like `nvim_create_autocmd`, pressing `<Leader>hh` on it immediately opens the help page. Combined with `C-]` to follow links within help and `C-t` to go back, you can deep-dive into any topic without typing.
**Tool anchor:** `:help`, `:help help-summary`, `:help notation`
**Drill:** (1) Open `:help usr_01` (the first user manual chapter) and read the table of contents. (2) Use `C-]` on a highlighted tag to follow it, then `C-t` to return. (3) Use `<Leader>fh` to search for "text-objects" and open the help page. (4) In any config file, place cursor on `vim.keymap.set` and press `<Leader>hh` to look it up. (5) Learn the naming conventions: try `:help i_CTRL-O` (insert-mode C-o), `:help v_d` (visual-mode delete), `:help 'number'` (the number option with quotes). (6) Use `:helpgrep pattern` to search within all help files.
**Tags:** help, help-navigation, help-tags, C-bracket, documentation, self-learning

## intermediate

### Alternate file
`C-^` (or `:b#`) switches between the current buffer and the alternate buffer (shown as `#` in `:ls`). The alternate buffer is typically the last file you were editing. Your config maps `<Leader>bb` to `:e #` for the same purpose. This single-keystroke toggle is the fastest way to flip between two related files: a header and its source, a module and its test, an implementation and its caller. Building the habit of using `C-^` eliminates the overhead of Telescope or buffer lists for the most common two-file workflow.
**Key concepts:** C-^, :b#, alternate buffer, #a flag in :ls, two-file toggle, <Leader>bb
**Tip:** The alternate file workflow is ideal for C++ header/source pairs: edit `decoder.h`, switch to `decoder.cpp` with `:e decoder.cpp`, and now `C-^` toggles between them. Your `<Leader>bb` mapping does the same thing. For test-driven development, keep the test file as the alternate: open `decoder.cpp`, then `:e decoder_test.cpp`, and now `C-^` toggles between implementation and test.
**Tool anchor:** `:help alternate-file`, `:help CTRL-^`, `:help :b#`
**Drill:** Open a C++ source file and its header. (1) Use `C-^` to toggle between them 5 times rapidly. (2) Now open a third file -- note that the alternate is now the second file, not the first. (3) Run `:ls` and identify the `#` flag showing the alternate. (4) Use `<Leader>bb` to toggle -- verify it does the same thing as `C-^`. (5) Practice the workflow: open implementation, open test, then `C-^` to toggle between them while coding. Compare the speed to using `<Leader>fb` for the same toggle.
**Tags:** alternate-file, C-caret, buffer-toggle, header-source, two-file-workflow

### Telescope pickers
Beyond file finding and grep, Telescope provides pickers for nearly everything in Neovim. Your config maps: `<Leader>fh` for help tags, `<Leader>fm` for man pages, `<Leader>f'` for marks, `<Leader>fr` for registers, `<Leader>fk` for keymaps, `<Leader>fd` for diagnostics, `<Leader>fc` for Neovim config files, and `<Leader>fp` for plugin source files. LSP pickers are bound on attach: `gd` for definitions, `grr` for references, `gri` for implementations, `gO` for document symbols, `gW` for workspace symbols. Each picker replaces a tedious manual lookup with instant fuzzy search.
**Key concepts:** specialized pickers, help_tags, man_pages, marks, registers, keymaps, diagnostics, LSP pickers, resume
**Tip:** `<Leader>fk` (keymaps picker) is invaluable when you forget a mapping: type part of the description or the keys themselves, and Telescope shows all matching keymaps with their source. When learning your config, use this constantly. `<Leader>fd` (diagnostics) gives a project-wide view of all LSP errors and warnings, sortable and filterable -- often more useful than scrolling through `trouble.nvim` for a quick overview.
**Tool anchor:** `:help telescope.builtin`, `:help telescope.builtin.keymaps`
**Drill:** (1) Use `<Leader>fk` and search for "split" to find all split-related keymaps. (2) Use `<Leader>fd` to see all diagnostics in the workspace and jump to one. (3) Use `<Leader>fh` and search for "text-objects" to find the help topic. (4) Use `<Leader>fr` to browse registers and paste one into the buffer. (5) Use `<Leader>fc` to navigate your Neovim config files. (6) Use `<Leader>f'` to jump to a previously set mark. Practice using Telescope as your primary discovery and navigation tool.
**Tags:** telescope-pickers, help-tags, keymaps, diagnostics, marks, registers, LSP-pickers

### Quickfix fundamentals
The quickfix list is a global list of file positions (filename, line, column, description). `:copen` opens it in a window, `:cnext`/`:cprev` (or your `]q`/`[q` mappings) navigate through entries, and `:cdo {cmd}` executes a command on each entry. The quickfix list is populated by commands like `:grep`, `:make`, `:vimgrep`, and LSP operations. Your trouble.nvim config provides a `<Leader>lq` mapping that shows the quickfix list in a trouble.nvim panel with better formatting and folding.
**Key concepts:** :copen, :cnext/:cprev, ]q/[q, :cdo, populating with :grep/:make, quickfix as location list
**Tip:** The fastest way to populate the quickfix list in your setup is `:grep pattern` (which uses `rg` since your `grepprg = "rg --vimgrep"`). After `:grep TODO`, the quickfix fills with every TODO in your project. Navigate with `]q`/`[q` to jump through them, or `<Leader>lq` to see them all in trouble.nvim. For batch operations, `:cdo s/old/new/ | update` applies a substitution to every quickfix entry and saves.
**Tool anchor:** `:help quickfix`, `:help :copen`, `:help :cdo`
**Drill:** (1) Run `:grep TODO` to populate the quickfix with all TODOs in your project. (2) Use `:copen` to view the list, then `]q`/`[q` to navigate through entries. (3) Open the same list in trouble.nvim with `<Leader>lq` and compare the UI. (4) Run `:grep "include.*vector"` and use `:cdo s/vector/array/ | update` to replace across all matching files (undo with `:cdo u`). (5) Use `:make` to populate the quickfix with compiler errors and navigate to each.
**Tags:** quickfix, copen, cnext, cprev, cdo, grep, trouble

### Location list
The location list is a per-window version of the quickfix list. Each window has its own location list, opened with `:lopen`, navigated with `:lnext`/`:lprev`, and operated on with `:ldo`. Location lists are populated by window-specific operations like `:lvimgrep` and some LSP commands. Your trouble.nvim config maps `<Leader>ll` to show the location list in a trouble panel. Use the location list when results are relevant only to the current context, reserving the quickfix for project-wide results.
**Key concepts:** :lopen, :lnext/:lprev, per-window scope, :ldo, location vs quickfix, window-local
**Tip:** A practical distinction: use the quickfix list (`:grep`) for project-wide searches and the location list (`:lvimgrep`) for file-local searches. For example, `:lvimgrep /TODO/ %` finds TODOs in the current file only, keeping your quickfix list free for the project-wide search results. Your trouble.nvim `<Leader>ll` shows the location list, while `<Leader>lq` shows the quickfix -- use both simultaneously for different scopes.
**Tool anchor:** `:help location-list`, `:help :lopen`, `:help :ldo`
**Drill:** (1) In a C++ file, run `:lvimgrep /TODO/ %` to find all TODOs in the current file. (2) Open with `:lopen` and navigate with `:lnext`/`:lprev`. (3) In a different window, run a different `:lvimgrep` -- verify each window has its own location list. (4) Open trouble.nvim with `<Leader>ll` for the location list. (5) Meanwhile, run `:grep TODO` to populate the global quickfix. (6) Verify you now have both: `<Leader>lq` for project-wide TODOs (quickfix) and `<Leader>ll` for file-local TODOs (location list).
**Tags:** location-list, per-window, lopen, lnext, lprev, quickfix-vs-loclist

### Argument list
The argument list (arglist) is a curated subset of buffers you want to operate on. `:args *.cpp` loads all matching files, `:argdo {cmd}` runs a command on each, `:argadd` adds the current file, and `:argdelete %` removes it. Unlike the buffer list (which grows as you open files), the arglist is explicitly constructed. `:args` without arguments shows the current list. The arglist is particularly useful for batch operations across a specific set of files without affecting all open buffers.
**Key concepts:** :args, :argdo, :argadd/:argdelete, argument vs buffer list, batch operations, explicit file set
**Tip:** For a targeted refactoring across SBE decoder files: `:args src/sbe/**/*.cpp` loads all SBE source files into the arglist, then `:argdo %s/old_name/new_name/ge | update` renames across exactly those files. The `e` flag suppresses errors when a file has no matches, and `| update` saves each file. This is more controlled than `:bufdo` which operates on every open buffer (including unrelated files).
**Tool anchor:** `:help arglist`, `:help :args`, `:help :argdo`
**Drill:** (1) Run `:args src/*.cpp` to load C++ source files into the arglist. (2) Run `:args` to see the list and identify the current file (marked with brackets). (3) Use `:next`/`:prev` to navigate through arglist files. (4) Run `:argdo %s/copyright 2025/copyright 2026/ge | update` to update copyright year across all files. (5) Use `:argadd test/*.cpp` to add test files to the list. (6) Compare `:argdo` with `:bufdo` -- open an unrelated file and verify `:bufdo` affects it but `:argdo` does not.
**Tags:** argument-list, argdo, args, batch-refactoring, file-set, argadd

### Trouble.nvim
Trouble.nvim provides a structured panel view for diagnostics, quickfix, location lists, LSP references, and TODO comments. Your config maps: `<Leader>dd` for buffer diagnostics (filtered to current buffer), `<Leader>dw` for workspace diagnostics (all files), `<Leader>df` for a diagnostic float on the current line, `<Leader>dt` to toggle diagnostics on/off, `<Leader>lq` for quickfix, `<Leader>ll` for location list, and `<Leader>lt` for TODO comments. The panel opens with `focus = true`, meaning it immediately receives cursor focus for navigation.
**Key concepts:** diagnostics panel, workspace vs buffer, quickfix view, location list view, TODO list, focus mode, toggle
**Tip:** `<Leader>dd` (buffer diagnostics) is your first stop when fixing a single file -- it shows only the current buffer's errors and warnings in a navigable list. `<Leader>dw` (workspace diagnostics) is for broader triage, showing issues across the entire project from clangd, pyright, lua_ls, and rust-analyzer. The `toggle` action means pressing the same key again closes the panel, making it easy to pop open for a quick check and dismiss.
**Tool anchor:** `:help trouble.nvim`, `<Leader>dd`, `<Leader>dw`

### Navigation-heavy workflow patterns
An efficient navigation workflow chains multiple techniques: `gd` (definition) to understand a function, `grr` (references) to see callers, send references to quickfix for systematic review, `]q`/`[q` to visit each caller, `g;` to return to the last edit, and `C-o` to unwind the entire navigation chain. Mastering when to use each eliminates wasted navigation.
**Key concepts:** definition-to-references-to-quickfix cycle, navigation scope selection, jump vs change vs tag stack, systematic code reading
**Tip:** When understanding a new C++ codebase (e.g., ramping on CME MDP 3.0 integration), use this workflow: `gd` on the main entry point, read the function, `gd` on each called function to drill deeper, use `C-o` to back out when done. For systematic review: `grr` on a key function, send results to quickfix (Telescope action), then `]q`/`[q` through all callers. Set marks on important locations (`mA`, `mB`) as you discover them. This structured approach replaces random file-opening.
**Tool anchor:** `:help gd`, `:help grr`, `:help jumplist`, `:help changelist`
**Drill:** Pick a function you want to understand deeply in your codebase. (1) Use `gd` to go to its definition. (4) Use `grr` on the function to find all callers. (5) In the Telescope picker, use `C-q` to send all results to quickfix. (6) Navigate callers with `]q`/`[q`, using `zz` after each jump. (7) Use `g;` to return to your last edit. Map out the navigation stacks you used.
