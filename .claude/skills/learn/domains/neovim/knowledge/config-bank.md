## beginner

### Neovim Lua basics -- require, modules, vim.opt vs vim.o
Neovim embeds LuaJIT as its scripting runtime, and all modern configuration is written in Lua rather than Vimscript. The `require("module")` function loads Lua modules from `lua/` directories on the runtimepath, caching them after first load. Your config uses `require("custom.core")` to load `lua/custom/core/init.lua`, where dots map to directory separators. For setting options, Neovim provides several interfaces: `vim.opt` returns option objects with list/map methods (`:append()`, `:prepend()`, `:remove()`), `vim.o` provides direct string access to global options, `vim.bo` and `vim.wo` access buffer-local and window-local options respectively, and `vim.g` accesses global variables (like `vim.g.mapleader = " "`). Understanding which interface to use matters because `vim.opt.shortmess:append("W")` works but `vim.o.shortmess = vim.o.shortmess .. "W"` is the equivalent raw form.
**Key concepts:** require, module caching, lua/ directory, vim.opt, vim.o, vim.bo, vim.wo, vim.g, option objects, runtimepath
**Tip:** `vim.opt` returns a special object, not the value itself. To get the actual value, use `vim.opt.tabstop:get()` or `vim.o.tabstop`. This trips up beginners who write `if vim.opt.number then` (always truthy because the object exists) instead of `if vim.o.number then` or `if vim.opt.number:get() then`.
**Tool anchor:** `:lua vim.print(vim.opt.completeopt:get())` to inspect option values
**Drill:** In your Neovim command line, run `:lua vim.print(type(vim.opt.tabstop))` and `:lua vim.print(type(vim.o.tabstop))`. Notice the first is "table" (option object) and the second is "number" (raw value). Then try `:lua vim.opt.wildmode = "longest:full,full"` and compare with `:lua vim.print(vim.opt.wildmode:get())` to see how string options with commas become lists. Experiment with `:lua vim.opt.shortmess:append("S")` and verify with `:set shortmess?`.
**Tags:** lua, require, vim-opt, vim-o, vim-g, modules, options, basics

### Config directory structure -- init.lua and the lua/ tree
Neovim's configuration lives in `~/.config/nvim/` with a specific directory structure. `init.lua` is the entry point, loaded on startup. The `lua/` directory contains Lua modules loadable via `require()`. Your config has `lua/custom/core/` (options, keymaps, autocmds), `lua/custom/plugins/` (16 lazy.nvim spec files), `lua/custom/utils/` (helpers), `lua/custom/health.lua` (checkhealth module), and `lua/custom/drill.lua` (the drill harness). The `after/` directory (for ftplugin overrides) and `ftplugin/` (filetype-specific settings) load after all other config. The `plugin/` directory is for automatically-loaded scripts. This structure is not arbitrary: `require("custom.plugins.telescope")` maps to `lua/custom/plugins/telescope.lua`, and lazy.nvim's `{ import = "custom.plugins" }` loads every file in that directory as a plugin spec.
**Key concepts:** init.lua, lua/ directory, after/, ftplugin/, plugin/, require path mapping, lazy.nvim import, filetype.lua, runtimepath order
**Tip:** Your `init.lua` loads in a specific order: `require("custom.core")` first (setting leader key and options before any plugin loads), then lazy.nvim bootstrap and setup, then `require("custom.utils.globals")` last. This ordering matters because `vim.g.mapleader` must be set before any keymap uses `<Leader>`, and lazy.nvim must be on the runtimepath before `require("lazy")` works.
**Tool anchor:** `:lua vim.print(vim.fn.stdpath("config"))` to find your config directory
**Drill:** Map out your config's loading order by adding `vim.notify("Loading: core/init")` at the top of each file. Start Neovim and read the messages in order with `:messages`. Then create a new file at `lua/custom/plugins/test.lua` that returns `{ "test/test.nvim" }` and observe that lazy.nvim's `import` picks it up. Remove the test file when done.
**Tags:** config, directory-structure, init-lua, require, after, ftplugin, loading-order

### vim.keymap.set -- modes, options, and modern mappings
`vim.keymap.set(mode, lhs, rhs, opts)` is the standard API for creating keymaps. The first argument is a mode string or table: `"n"` (normal), `"i"` (insert), `"v"` (visual+select), `"x"` (visual only), `"o"` (operator-pending), `"s"` (select), or a table like `{"n", "x", "o"}` for multiple modes. Your config demonstrates key patterns: `expr = true` for conditional mappings (the j/k wrapped-line handling), `buffer = event.buf` for buffer-local LSP keymaps, `desc = "..."` for which-key integration, `remap = true` to allow recursive mapping (window navigation), and `silent = true` to suppress command-line echo. The `rhs` can be a string (Vim command) or a Lua function, and functions receive no arguments by default.
**Key concepts:** vim.keymap.set, mode strings, opts table, desc, buffer, silent, expr, remap, Lua function rhs, which-key integration
**Tip:** Always provide a `desc` for your keymaps -- your which-key plugin uses these descriptions to build the popup menu. Without `desc`, the keymap still works but appears as an unlabeled entry. Your LSP keymaps prefix descriptions with "LSP: " for clear categorization in which-key.
**Tool anchor:** `<Leader>fk` to search all keymaps via Telescope, `:map <Leader>` to list leader mappings
**Drill:** Create a temporary keymap: `:lua vim.keymap.set("n", "<Leader>tt", function() vim.notify("test") end, { desc = "Test keymap" })`. Verify it appears in `<Leader>fk` (Telescope keymaps) with the description. Then create a buffer-local keymap: `:lua vim.keymap.set("n", "<Leader>tb", function() vim.notify("buffer only") end, { buffer = 0, desc = "Buffer test" })`. Switch to another buffer and verify it does not work there. Delete both with `vim.keymap.del`.
**Tags:** keymaps, vim-keymap-set, modes, opts, desc, buffer-local, expr, which-key

### Options deep dive -- vim.opt list and map methods
`vim.opt` returns Option objects that understand list-style and map-style Vim options. Your config uses `vim.opt.rtp:prepend(lazypath)` to add lazy.nvim to the front of the runtimepath, `vim.opt.shortmess:append({ W = true, I = true, c = true, C = true })` to add shortmess flags, `vim.opt.sessionoptions = { ... }` to set a list option, and `vim.opt.fillchars = { ... }` to set a map option. The `:get()` method returns the current value as a Lua type (table for lists/maps, string/number/boolean for simple options). The `:remove()` method removes items from list or map options. Understanding the difference between set (`:=`), append (`:append`), prepend (`:prepend`), and remove (`:remove`) is essential because these operations work differently than plain assignment on list/map options.
**Key concepts:** vim.opt objects, :get(), :append(), :prepend(), :remove(), list options (rtp, wildignore), map options (fillchars, listchars), set vs append
**Tip:** `vim.opt.fillchars = { fold = " " }` replaces the entire fillchars value, losing any defaults. To add without replacing, use `vim.opt.fillchars:append({ fold = " " })`. Your config intentionally replaces all fillchars with a curated set, which is correct for your use case but could be surprising when copying individual settings to a different config.
**Tool anchor:** `:lua vim.print(vim.opt.fillchars:get())` to inspect map options, `:set fillchars?` for the Vim representation
**Drill:** Run `:lua vim.print(vim.opt.rtp:get())` to see the full runtimepath as a Lua table. Then try `:lua vim.opt.rtp:append("/tmp/test")` and verify the path appears at the end. Remove it with `:lua vim.opt.rtp:remove("/tmp/test")`. Experiment with `:lua vim.opt.shortmess:remove("c")` and `:set shortmess?` to see the effect. Compare `vim.opt.completeopt:get()` (returns a table) with `vim.o.completeopt` (returns a string).
**Tags:** vim-opt, options, list-options, map-options, append, prepend, remove, fillchars

### Plugin management with lazy.nvim -- spec format and basics
lazy.nvim is your plugin manager, bootstrapped in init.lua by cloning the git repository to the data directory if not present. Plugin specs are Lua tables with the short-form `{ "owner/repo" }` plus optional configuration: `config` (function called after loading), `opts` (table passed to the plugin's `setup()` function), `dependencies` (plugins loaded first), `build` (command run after install/update), and `lazy = false` (load immediately instead of lazily). Your config uses `{ import = "custom.plugins" }` to auto-load every file in `lua/custom/plugins/` as a plugin spec, meaning each file returns a table (or list of tables) that lazy.nvim processes. The `checker = { enabled = true, notify = false }` setting silently checks for plugin updates.
**Key concepts:** lazy.nvim bootstrap, plugin spec, config function, opts table, dependencies, build command, import directory, checker, rocks support
**Tip:** The `{ import = "custom.plugins" }` pattern means adding a new plugin is just creating a new file in `lua/custom/plugins/`. No central registry to edit. But be careful: every `.lua` file in that directory is loaded as a spec, so utility or helper files should live elsewhere (like your `lua/custom/utils/`).
**Tool anchor:** `<Leader>L` to open the Lazy UI, `:Lazy profile` to see startup timing
**Drill:** Open the Lazy UI with `<Leader>L`. Press `P` to see the profile (startup times per plugin). Note which plugins are lazy-loaded vs loaded at startup. Then create a minimal test plugin spec at `lua/custom/plugins/test-plugin.lua`: `return { "echasnovski/mini.surround", opts = {} }`. Restart Neovim and verify it appears in `:Lazy`. Remove the file and run `:Lazy clean` to uninstall.
**Tags:** lazy-nvim, plugin-management, spec, config, opts, dependencies, import, bootstrap

### vim.api.nvim_create_autocmd -- events, patterns, callbacks
Autocommands execute code in response to events. `vim.api.nvim_create_autocmd(event, opts)` creates one, where `event` is a string or table of event names, and `opts` includes: `pattern` (filename pattern like `*.cpp`), `group` (augroup for organization/clearing), `callback` (Lua function receiving an event table), `once` (fire once then delete), and `command` (Vimscript string alternative to callback). Your config's autocmds.lua creates a "close_with_esc" group that maps Escape to close helper buffers (lazy, help, lspinfo, man, qf, oil, checkhealth). The augroup pattern `vim.api.nvim_create_augroup("name", { clear = true })` ensures re-sourcing the config does not create duplicate autocmds because `clear = true` deletes existing autocmds in the group first.
**Key concepts:** nvim_create_autocmd, events, pattern, group, callback, once, nvim_create_augroup, clear, event table, BufEnter, FileType, BufWritePost
**Tip:** The callback function receives a table with `event` (event name), `buf` (buffer number), `file` (filename), `match` (matched pattern), and `data` (event-specific data, like `client_id` for LspAttach). Use this table instead of `vim.api.nvim_get_current_buf()` because autocmds can fire for non-current buffers.
**Tool anchor:** `:autocmd custom-*` to list your custom augroups, `:lua vim.print(vim.api.nvim_get_autocmds({ group = "custom-close_with_esc" }))` to inspect
**Drill:** Create a temporary autocmd that prints the filename every time you enter a buffer: `:lua vim.api.nvim_create_autocmd("BufEnter", { group = vim.api.nvim_create_augroup("test-autocmd", { clear = true }), callback = function(ev) vim.notify("Entered: " .. ev.file) end })`. Switch between buffers and observe. Then clear it: `:lua vim.api.nvim_del_augroup_by_name("test-autocmd")`. Verify it stopped firing.
**Tags:** autocmd, events, patterns, augroup, callback, BufEnter, FileType, clear

### :help discipline -- navigating the built-in documentation
Neovim's `:help` system is the authoritative reference and is far more comprehensive than any web resource. `:help vim.lsp.buf.hover` jumps directly to the API docs; `<C-]>` follows a tag link; `<C-t>` pops back; `:helpgrep pattern` searches across all help files. Help topics follow naming conventions: `vim.api.*` for the API, `vim.lsp.*` for LSP, `vim.treesitter.*` for treesitter, options are their name (`:help tabstop`), events are their name (`:help BufEnter`). Your config maps `<Leader>hh` to `:help <cword>` (help for word under cursor) and `<Leader>hH` for WORD under cursor, plus `<Leader>hm`/`<Leader>hM` for man pages, making it fast to look up anything without leaving the editor.
**Key concepts:** :help, C-], C-t, :helpgrep, help tags, :help api, :help lsp, naming conventions, <Leader>hh, man pages
**Tip:** When you encounter an unfamiliar Neovim function in someone's config, place your cursor on it and press `<Leader>hh`. For `vim.api.nvim_create_autocmd`, it jumps straight to the API reference. For Vim options, just `:help` the option name. This is faster than searching the web and always matches your exact Neovim version.
**Tool anchor:** `<Leader>hh` for help on word under cursor, `<Leader>fh` for Telescope help tags search
**Drill:** Open your lspconfig.lua and place your cursor on `nvim_create_autocmd`. Press `<Leader>hh` to jump to the help. Read the signature and parameter documentation. Use `<C-]>` on any linked tag (like `autocmd-events`) to follow it. Use `<C-t>` to pop back. Then try `:helpgrep vim.lsp.config` to search for all mentions of the config API. Navigate the quickfix list with `]q`/`[q`.
**Tags:** help, documentation, tags, helpgrep, navigation, discipline, reference

### vim.fn vs vim.api vs vim.cmd -- when to use each
Neovim provides three interfaces to its functionality, each with different use cases. `vim.api` (`nvim_*` functions) is the primary Lua API -- pure Lua, fast, type-safe, the preferred choice for new code. `vim.fn` accesses Vimscript functions (`vim.fn.expand`, `vim.fn.executable`, `vim.fn.system`) -- necessary for Vimscript built-ins that have no API equivalent yet. `vim.cmd` executes Ex commands as strings (`vim.cmd("edit file.txt")`, `vim.cmd.split()`) -- useful for commands with complex syntax or when calling plugin commands. Performance differences matter in hot paths: `vim.api` calls are direct C function calls, `vim.fn` goes through the Vimscript evaluation layer (slower), and `vim.cmd` parses a string (slowest). Your config uses all three appropriately: `vim.api` for autocmds and keymaps, `vim.fn` for `expand`, `executable`, `stdpath`, and `vim.cmd` for `split`, `tabclose`, `resize`.
**Key concepts:** vim.api, vim.fn, vim.cmd, performance differences, Vimscript bridge, vim.cmd.split() method syntax, direct C calls
**Tip:** When a `vim.fn` function has a `vim.api` equivalent, prefer the API version. For example, use `vim.api.nvim_buf_get_name(0)` instead of `vim.fn.bufname()`, and `vim.api.nvim_get_current_buf()` instead of `vim.fn.bufnr()`. The API versions are faster and return proper Lua types instead of Vimscript types that may need conversion.
**Tool anchor:** `:lua vim.print(vim.fn.stdpath("data"))` for fn usage, `:lua vim.print(vim.api.nvim_get_current_buf())` for api usage
**Drill:** Benchmark the three interfaces: `:lua local t = vim.uv.hrtime(); for i=1,10000 do vim.api.nvim_get_current_buf() end; print((vim.uv.hrtime()-t)/1e6 .. "ms")`. Repeat with `vim.fn.bufnr()` and `vim.cmd("echo bufnr()")`. Compare the times. Then try calling a Vimscript function that has no API equivalent: `:lua vim.print(vim.fn.getcharstr())` -- this must use `vim.fn` because there is no `vim.api` equivalent for reading a single character.
**Tags:** vim-fn, vim-api, vim-cmd, performance, vimscript-bridge, interface-choice

### Error handling and debugging -- pcall, vim.inspect, notifications
Lua error handling in Neovim uses `pcall(fn, args)` (protected call) which returns `ok, result` instead of throwing. Your config uses pcall in critical paths: `pcall(vim.treesitter.start)` in the treesitter FileType autocmd (silently fails if no parser exists for the filetype), `pcall(vim.api.nvim_win_close, win, true)` in the drill harness (ignores errors if the window was already closed), and `pcall(vim.api.nvim_buf_set_name, buf, name)` (ignores duplicate name errors). For debugging, `vim.inspect(value)` produces a human-readable string representation of any Lua value, and `vim.print(value)` is shorthand for `print(vim.inspect(value))`. Your `_G.P = function(v) print(vim.inspect(v)); return v end` global in utils/globals.lua provides quick inspection. The `:messages` command shows notification history, and `:verbose` prefixed to commands shows where settings were defined.
**Key concepts:** pcall, vim.inspect, vim.print, vim.notify, :messages, :verbose, _G.P helper, error levels, notification system
**Tip:** When debugging a failing autocmd or plugin config, wrap the body in pcall and log the error: `local ok, err = pcall(function() ... end); if not ok then vim.notify("Error: " .. err, vim.log.levels.ERROR) end`. This prevents one failing autocmd from breaking your entire config, which is why your treesitter FileType autocmd uses `pcall(vim.treesitter.start)` instead of bare `vim.treesitter.start()`.
**Tool anchor:** `:messages` to see notification history, `:verbose set tabstop?` to trace option origin, `P(value)` for quick inspection
**Drill:** Create a deliberate error and handle it: `:lua local ok, err = pcall(vim.api.nvim_buf_get_lines, 99999, 0, 1, true); vim.print(ok, err)`. The first return is `false`, the second is the error message. Then use `P()` to inspect a complex object: `:lua P(vim.lsp.get_clients())`. Check where an option was set: `:verbose set updatetime?`. Finally, examine the notification history after triggering some events: `:messages`.
**Tags:** error-handling, pcall, vim-inspect, debugging, messages, verbose, notifications

The health check is the fastest way to diagnose why LSP or formatting is not working.

### lazy.nvim advanced -- event, cmd, ft, keys triggers and config vs opts
Lazy loading defers plugin loading until a trigger fires, reducing startup time. lazy.nvim supports several trigger types: `event = "BufReadPre"` (load on event), `cmd = "Trouble"` (load when command is run), `ft = "python"` (load for filetype), `keys = { { "<Leader>dd", ... } }` (load on keypress). Your config uses multiple strategies: Trouble loads on `cmd = "Trouble"` with keys that reference the command, conform loads on `event = { "BufReadPre", "BufNewFile" }`, treesitter loads immediately with `lazy = false`. The `config` function runs arbitrary Lua after loading, while `opts` is a table passed to the plugin's `.setup()` method (sugar for `config = function(_, opts) require("plugin").setup(opts) end`). The `init` function runs at startup before the plugin loads, useful for setting globals.
**Key concepts:** lazy loading, event trigger, cmd trigger, ft trigger, keys trigger, config vs opts vs init, priority, cond, build step, startup time
**Tip:** Your Trouble plugin uses `keys = { { "<leader>dd", "<cmd>Trouble diagnostics toggle filter.buf=0<CR>", desc = "..." } }` which both defines the trigger and the mapping. When you press `<Leader>dd` before Trouble is loaded, lazy.nvim loads Trouble first, then executes the mapping. This is seamless to the user but saves startup time by not loading Trouble until needed.
**Tool anchor:** `:Lazy profile` to see which plugins are lazy vs loaded, `:Lazy log` for update history
**Drill:** Check your startup time with `:Lazy profile` and identify the slowest-loading plugins. Pick a plugin that loads with `lazy = false` (like treesitter) and experiment with adding `event = "BufReadPost"` instead. Measure the startup difference. Then examine the difference between `opts = { focus = true }` (Trouble) and `config = function() ... end` (conform). Try converting one to the other and verify it works identically.
**Tags:** lazy-nvim, lazy-loading, event, cmd, ft, keys, config-vs-opts, startup-time, priority

### vim.api essentials -- buffers, highlights, commands, windows
The `vim.api` module exposes Neovim's C API to Lua, providing the foundational building blocks for all config customization. Key functions you use daily: `nvim_buf_set_keymap` / `nvim_buf_get_lines` / `nvim_buf_set_lines` for buffer manipulation, `nvim_set_hl` for highlight group definition, `nvim_create_user_command` for custom commands, `nvim_get_current_buf` / `nvim_get_current_win` for context. Your config uses these throughout: `nvim_create_augroup` and `nvim_create_autocmd` for autocmds, `nvim_set_hl` would be used for color overrides, `nvim_open_win` in the drill harness for floating windows. The API follows a naming convention: `nvim_<object>_<action>` where object is buf, win, tabpage, or omitted for global.
**Key concepts:** nvim_buf_*, nvim_win_*, nvim_set_hl, nvim_create_user_command, nvim_get_current_buf, nvim_open_win, naming convention, buffer/window/tabpage model
**Tip:** `vim.api` functions use 0-based indexing for buffers (0 = current buffer) and 0-based line/column indexing in most APIs, but `nvim_win_set_cursor` uses 1-based rows and 0-based columns. This inconsistency is the most common source of off-by-one bugs in Neovim Lua code. Always check the help for each function.
**Tool anchor:** `:lua vim.print(vim.api.nvim_get_current_buf())` for current buffer number, `:lua vim.print(vim.api.nvim_buf_get_lines(0, 0, -1, false))` for all lines
**Drill:** Get the current buffer number, then read its lines: `:lua local buf = vim.api.nvim_get_current_buf(); vim.print(vim.api.nvim_buf_get_lines(buf, 0, 5, false))`. Create a scratch buffer: `:lua local b = vim.api.nvim_create_buf(false, true); vim.api.nvim_buf_set_lines(b, 0, -1, false, {"hello", "world"}); vim.cmd("split"); vim.api.nvim_win_set_buf(0, b)`. Create a custom command: `:lua vim.api.nvim_create_user_command("Hello", function() vim.notify("Hello!") end, {})`. Test it with `:Hello`.
**Tags:** vim-api, buffers, windows, highlights, commands, nvim-buf, nvim-win, nvim-set-hl

### Autocommand patterns -- augroup, BufEnter vs FileType, clearing
Understanding autocommand event timing is critical for correct config behavior. `BufEnter` fires when you switch to a buffer (even if already loaded), `BufReadPost` fires after reading a file into a buffer (once per read), `FileType` fires when filetype is set (once per buffer unless filetype changes), `BufWritePre` fires before writing (for format-on-save), and `LspAttach` fires when an LSP client attaches. Your config's autocmds use `FileType` with specific patterns (`lazy`, `help`, `qf`, etc.) to set up buffer-local keymaps for utility windows. The augroup with `{ clear = true }` is essential for idempotent config: without it, re-sourcing your config (`:so %`) creates duplicate autocmds that fire multiple times.
**Key concepts:** event timing, BufEnter vs BufReadPost vs FileType, once, augroup clearing, re-source safety, event ordering, nested
**Tip:** Use `FileType` for per-language settings (buffer-local options, keymaps) and `BufReadPost`/`BufNewFile` for file-processing tasks (linting, formatting setup). Do not use `BufEnter` for one-time setup because it fires every time you switch windows, even within the same buffer. Your nvim-lint uses `BufWritePost` correctly because linting should run after the file is saved, not while editing.
**Tool anchor:** `:autocmd FileType` to list all FileType autocmds, `:verbose autocmd BufWritePost` to see which file defined each autocmd
**Drill:** Create an autocmd that logs event timing: `:lua for _, ev in ipairs({"BufEnter", "BufReadPost", "FileType", "BufWinEnter"}) do vim.api.nvim_create_autocmd(ev, { group = vim.api.nvim_create_augroup("test-events", { clear = true }), callback = function(e) vim.notify(e.event .. " " .. (e.file or "")) end }) end`. Open a file and observe the order. Switch buffers and note which events fire again. Clean up with `:lua vim.api.nvim_del_augroup_by_name("test-events")`.
**Tags:** autocmd, augroup, BufEnter, FileType, BufReadPost, clearing, event-timing, re-source

### Filetype plugins -- after/ftplugin and buffer-local settings
The `after/ftplugin/<ft>.lua` pattern lets you set buffer-local options and keymaps that apply only to specific filetypes. These files are sourced after all other filetype detection and plugin loading, making them the ideal place for per-language overrides. For example, `after/ftplugin/cpp.lua` could set `vim.bo.commentstring = "// %s"`, `vim.bo.shiftwidth = 4` for a project with 4-space C++ indentation, or add buffer-local keymaps for C++-specific workflows. The `vim.bo` (buffer option) prefix ensures settings do not bleed into other filetypes. You can also use `ftplugin/<ft>.lua` (without `after/`) but `after/` guarantees your overrides are applied last.
**Key concepts:** after/ftplugin, buffer-local options, vim.bo, per-language settings, commentstring, shiftwidth, filetype detection, ftplugin loading order
**Tip:** For your C++ SBE development, create `after/ftplugin/cpp.lua` with `vim.bo.textwidth = 100` and `vim.opt_local.colorcolumn = "100"` to enforce your team's line length. Since these are buffer-local, they will not affect Lua files (which your config formats at 120 characters with stylua).
**Tool anchor:** `:verbose set shiftwidth?` to see where an option was last set, `:scriptnames` to list all sourced files
**Drill:** Create `~/.config/nvim/after/ftplugin/cpp.lua` with `vim.bo.commentstring = "// %s"` and `vim.notify("C++ ftplugin loaded")`. Open a C++ file and verify the notification appears and `gcc` uses `//` comments. Open a Python file and verify `commentstring` is different. ### Module caching and startup -- require mechanics and bytecode

Understanding this caching explains why changes to a Lua file require either restarting Neovim or manually clearing `package.loaded["module"]`. If your config changes are not being picked up, `vim.loader.reset()` clears the cache.

### Event loop and scheduling -- vim.schedule, vim.defer_fn, vim.on_key
Neovim's event loop is single-threaded: all UI updates, autocmds, and Lua callbacks run sequentially on the main loop. `vim.schedule(fn)` queues a function to run on the next main loop iteration, which is essential for calling Neovim API from libuv callbacks or other non-main-loop contexts. `vim.defer_fn(fn, ms)` schedules a function after a delay (implemented as a libuv timer + vim.schedule). `vim.on_key(fn, ns_id)` registers a callback that fires on every keypress, useful for logging, analytics, or custom key handling. Understanding this model explains why long-running Lua computations freeze the UI (they block the main loop) and why async operations require the schedule/callback pattern.
**Key concepts:** event loop, main loop, vim.schedule, vim.defer_fn, vim.on_key, single-threaded, blocking, libuv integration, callback queuing
**Tip:** If you need to perform a heavy computation (like parsing a large file) without freezing the UI, break it into chunks with `vim.defer_fn`: process N items, schedule the next batch with `vim.defer_fn(next_batch, 0)`, which yields control to the main loop between batches. This is a cooperative multitasking pattern that keeps the editor responsive.
**Tool anchor:** `:lua vim.defer_fn(function() vim.notify("delayed!") end, 2000)` to test deferred execution
**Drill:** Demonstrate the single-threaded nature: `:lua vim.notify("before"); for i=1,1e8 do end; vim.notify("after")` -- the UI freezes during the loop. For integration tests requiring buffers, windows, or LSP, plenary.test_harness launches a headless Neovim and runs tests inside it. Test files conventionally live in `tests/` or `test/` directories. CI/CD typically uses a GitHub Action that installs Neovim + dependencies and runs the test suite. Your drill.lua's parsing functions (parse_text_corpus, parse_motion_corpus, build_pool) are pure functions that could be unit-tested without a running Neovim.
**Key concepts:** busted, plenary.test_harness, describe/it/assert, nlua, headless Neovim, unit vs integration tests, CI, pure functions, test isolation
**Tip:** Start by testing pure functions extracted from your plugin logic. Your drill.lua's `days_ago()`, `strip_cursor()`, and parsing functions take inputs and return outputs with no side effects -- perfect unit test candidates. Write tests like `assert.equals(42, days_ago("2026-04-19", os.time({year=2026, month=5, day=31, hour=12})))`.
**Tool anchor:** `nvim --headless -c "PlenaryBustedDirectory tests/ {minimal_init = 'tests/init.lua'}"` for running tests

### Advanced vim.api -- floating windows, extmarks, nvim_exec_lua
Advanced vim.api usage enables building complex UI elements entirely in Lua. `nvim_open_win(buf, enter, config)` creates floating windows with precise positioning (your drill harness uses this for prompt floats with `relative = "editor"`, `border = "rounded"`, `style = "minimal"`). `nvim_buf_set_extmark(buf, ns, row, col, opts)` creates extmarks -- invisible anchored positions that can display virtual text, highlights, or signs. Extmarks survive text edits by moving with the text, making them ideal for annotations, inline hints, and diagnostic rendering. `nvim_exec_lua(code, args)` executes Lua code in a specific context, useful for remote plugins. `nvim_chan_send(chan, data)` sends data to terminal channels. The combination of floating windows + extmarks + async subprocess execution enables building full TUI applications inside Neovim.
**Key concepts:** nvim_open_win, floating windows, relative positioning, nvim_buf_set_extmark, extmarks, virtual text, namespaces, nvim_exec_lua, nvim_chan_send
**Tip:** Your drill harness demonstrates a production pattern for floating windows: create a scratch buffer (`nvim_create_buf(false, true)`), set its content, open a floating window anchored to the editor corner, and use `focusable = false` to prevent accidental focus. The `noautocmd = true` flag prevents FileType and other autocmds from firing for the float buffer, avoiding interference from your config's autocmds.
**Tool anchor:** `:lua local b = vim.api.nvim_create_buf(false,true); vim.api.nvim_buf_set_lines(b,0,-1,false,{"Hello!"}); vim.api.nvim_open_win(b,true,{relative="cursor",row=1,col=0,width=20,height=1,border="rounded"})` for a quick float
**Drill:** Create a floating window from scratch: make a buffer, add content, open it as a float centered on screen. Then add extmarks with virtual text: `:lua local ns = vim.api.nvim_create_namespace("test"); vim.api.nvim_buf_set_extmark(0, ns, 0, 0, { virt_text = {{"<-- extmark", "Comment"}}, virt_text_pos = "eol" })`. The virtual text appears at end-of-line. Clear with `:lua vim.api.nvim_buf_clear_namespace(0, ns, 0, -1)`. Build a simple notification float that auto-closes after 3 seconds using `vim.defer_fn`.
