Official reference: https://www.lpi.org/our-certifications/exam-201-202-objectives/

## 207.1 Basic DNS server configuration (weight: 3)

**Description:** Candidates should be able to configure BIND to function as an authoritative and as a recursive, caching-only DNS server. This objective includes the ability to manage a running server and configuring logging.

**Key Knowledge Areas:**
- BIND 9.x configuration files, terms and utilities
- Defining the location of the BIND zone files in BIND configuration files
- Reloading modified configuration and zone files
- Awareness of dnsmasq, djbdns and PowerDNS as alternate name servers

**Files, terms and utilities:**
- `/etc/named.conf`
- `/var/named/`
- `rndc`
- `named-checkconf`
- `kill`
- `host`
- `dig`

### Exam focus areas

- **`/etc/named.conf`** — main BIND config file. Key sections: `options { }` (directory, forwarders, allow-query, allow-recursion, listen-on), `zone "..." { }` (type master/slave/hint/forward, file, allow-transfer). On Debian/Ubuntu the file is `/etc/bind/named.conf` split into `named.conf.options`, `named.conf.local`, `named.conf.default-zones`.
- **`/var/named/`** — default zone file directory (set by `directory` in options). On Debian: `/var/cache/bind/`.
- **`rndc`** — remote name daemon control. Key commands: `rndc reload` (reload all zones), `rndc reload example.com` (reload specific zone), `rndc reconfig` (reload config, pick up new zones), `rndc status`, `rndc flush` (clear cache), `rndc dumpdb` (dump cache to file), `rndc querylog` (toggle query logging). Uses shared key from `rndc.conf` or `rndc-confgen`.
- **`named-checkconf`** — validates `named.conf` syntax before reloading. Exit 0 = OK. Use `-z` to also check zone files referenced in the config.
- **`dig`** — DNS lookup utility. Key flags: `dig @server domain type` (e.g., `dig @localhost example.com A`), `+short` (terse output), `+trace` (follow delegation from root), `+norecurse` (test authoritative response), `+dnssec` (request DNSSEC records), `-x IP` (reverse lookup). Output sections: QUESTION, ANSWER, AUTHORITY, ADDITIONAL.
- **`host`** — simple DNS lookup. `host domain [server]`. `-t type` for specific record type. Simpler output than dig.
- **Authoritative vs recursive** — authoritative server answers from its own zone data (aa flag set). Recursive/caching server queries other servers on behalf of clients. `recursion yes/no;` in options. Best practice: separate authoritative and recursive roles.
- **Logging** — `logging { channel ...; category ...; };` in named.conf. Categories: queries, security, default, general. Channels: syslog, file, stderr, null. `severity` levels: critical, error, warning, notice, info, debug, dynamic.
- **Alternate name servers** — dnsmasq: lightweight DNS forwarder + DHCP, reads `/etc/hosts`, used in small networks/libvirt. djbdns: tinydns + dnscache, uses `/service/` directory. PowerDNS: modular, supports database backends (MySQL, PostgreSQL, LDAP).

## 207.2 Create and maintain DNS zones (weight: 3)

**Description:** Candidates should be able to create a zone file for a forward or reverse zone and hints for root level servers. This objective includes setting appropriate values for records, adding hosts in zones and adding zones to the DNS. A candidate should also be able to delegate zones to another DNS server.

**Key Knowledge Areas:**
- BIND 9 configuration files, terms and utilities
- Utilities to request information from the DNS server
- Layout, content and file location of the BIND zone files
- Various methods to add a new host in the zone files, including reverse zones

**Files, terms and utilities:**
- `/var/named/`
- Zone file syntax
- Resource record formats (A, AAAA, MX, NS, SOA, PTR, CNAME, TXT)
- `named-checkzone`
- `named-compilezone`
- `masterfile-format`
- `dig`
- `nslookup`
- `host`

### Exam focus areas

- **SOA record** — `@ IN SOA ns1.example.com. admin.example.com. (serial refresh retry expire minimum)`. Serial: YYYYMMDDNN convention. Refresh: how often slave checks for updates. Retry: retry interval if refresh fails. Expire: slave stops answering if master unreachable. Minimum/negative TTL: how long to cache NXDOMAIN.
- **Record types** — A (IPv4), AAAA (IPv6), MX (mail, has priority number, must point to A/AAAA not CNAME), NS (nameserver delegation), PTR (reverse lookup, in in-addr.arpa/ip6.arpa zones), CNAME (alias, cannot coexist with other records for same name), TXT (text, used for SPF/DKIM/DMARC), SRV (service location).
- **Zone file syntax** — `$TTL` directive sets default TTL. `$ORIGIN` sets default domain suffix. `@` refers to current origin. Fully qualified names end with `.` (dot). Relative names get $ORIGIN appended. Comments start with `;`.
- **Reverse zones** — for 192.168.1.0/24: zone name `1.168.192.in-addr.arpa`. PTR records use last octet: `10 IN PTR host.example.com.` (trailing dot required). For IPv6: `ip6.arpa` with nibble format.
- **Zone delegation** — NS records in parent zone point to child nameservers. Glue records (A records in parent zone) needed when nameserver is within the delegated zone.
- **`named-checkzone`** — validates zone file syntax. Usage: `named-checkzone example.com /var/named/example.com.zone`. Checks SOA serial, record syntax, CNAME conflicts.
- **`named-compilezone`** — converts zone files between formats (text ↔ raw). Used with `masterfile-format raw;` for faster loading of large zones.
- **`nslookup`** — interactive/non-interactive DNS queries. `nslookup domain [server]`. `set type=MX` in interactive mode. Deprecated in favor of dig/host but still on exam.

## 207.3 Securing a DNS server (weight: 2)

**Description:** Candidates should be able to configure a DNS server to run as a non-root user and run in a chroot jail. This objective includes secure exchange of data between DNS servers.

**Key Knowledge Areas:**
- BIND 9 configuration files
- Configuring BIND to run in a chroot jail
- Split configuration of BIND using the forwarders statement
- Configuring and using transaction signatures (TSIG)
- Awareness of DNSSEC and basic tools
- Awareness of DANE and related records

**Files, terms and utilities:**
- `/etc/named.conf`
- `/etc/passwd`
- DNSSEC
- `dnssec-keygen`
- `dnssec-signzone`

### Exam focus areas

- **Chroot jail** — run BIND in `/var/named/chroot/` so even if compromised, attacker is contained. Named starts with `-t /var/named/chroot`. All config/zone files must be inside chroot. `/etc/passwd` must have `named` user entry (runs as non-root). The `-u named` flag drops privileges.
- **TSIG (Transaction Signatures)** — shared-key authentication between master/slave or for rndc. Generate key: `dnssec-keygen -a HMAC-SHA256 -b 256 -n HOST keyname`. Add `key "keyname" { algorithm hmac-sha256; secret "base64..."; };` to named.conf on both servers. Reference in `server` or `allow-transfer` statements. Secures zone transfers and dynamic updates.
- **Forwarders / split DNS** — `forwarders { 8.8.8.8; };` in options sends unresolved queries to upstream. `forward only;` means only use forwarders (never iterate). `forward first;` tries forwarders first, then iterates. Split DNS: use `view` statements to serve different answers to internal vs external clients based on source IP (`match-clients`).
- **DNSSEC** — adds cryptographic signatures to DNS records. Key types: KSK (Key Signing Key) signs DNSKEY records, ZSK (Zone Signing Key) signs zone data. `dnssec-keygen -a RSASHA256 -b 2048 -f KSK example.com` (KSK), `dnssec-keygen -a RSASHA256 -b 1024 example.com` (ZSK). `dnssec-signzone -o example.com example.com.zone` produces signed zone file. RRSIG, DNSKEY, DS, NSEC/NSEC3 record types. Validation: `dnssec-validation auto;` in options.
- **DANE (DNS-based Authentication of Named Entities)** — TLSA records associate TLS certificates with domain names via DNSSEC. Format: `_443._tcp.www.example.com. IN TLSA usage selector matching-type cert-data`. Reduces dependency on CAs.
- **`allow-transfer`** — restrict zone transfers to specific IPs/keys: `allow-transfer { 192.168.1.2; key "transfer-key"; };`. Default should be `none` for security.
